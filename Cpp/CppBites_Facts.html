C++ Bites: Facts

language history, design principles, and key characteristics

About C++ Track

1.0 Synopsis

C++ is a general-purpose, statically typed, compiled language that gives developers direct control over hardware while supporting high-level abstractions at zero overhead. It traces its roots to Bell Labs in 1979, has been standardized continuously since 1998, and remains central to systems programming, game engines, embedded software, and performance-critical applications. This page collects the facts every C++ developer should know before writing a line of code.

2.0 Language History

Bjarne Stroustrup began developing "C with Classes" at Bell Labs in 1979. The language was renamed C++ in 1983 (the increment operator symbolizing an improvement on C). The first commercial compiler, Cfront, translated C++ to C. Major standards milestones:

3.0 Design Principles

The design of C++ is guided by a small set of principles that have remained stable across all revisions:
  • Zero-overhead abstraction - you don't pay in runtime cost for language features you don't use. A class with no virtual functions has no vtable. An unused template instantiation generates no code.
  • Value semantics by default - objects are copied, not reference-shared, unless you explicitly request a reference or pointer. This makes reasoning about ownership straightforward.
  • RAII - Resource Acquisition Is Initialization. Resources are tied to object lifetime so the compiler guarantees cleanup, even through exceptions.
  • Static type safety - type errors are caught at compile time, not runtime. Templates and concepts (C++20) extend this to generic code.
  • Direct hardware control - pointers, manual memory layout, intrinsics, and inline assembly are available when needed.
  • Multi-paradigm - C++ does not force a single style. Procedural, object-oriented, generic, and functional patterns all have first-class support.

4.0 Key Characteristics

A concise profile of C++ as a language:
  • Statically typed - every expression has a type known at compile time.
  • Compiled to native code - no intermediate bytecode; the compiler emits machine instructions directly.
  • No garbage collector - memory management is deterministic and explicit, controlled by constructors and destructors.
  • Smart pointers - std::unique_ptr and std::shared_ptr automate ownership without a GC.
  • Multiple inheritance - supported, with virtual inheritance to resolve the diamond problem.
  • Templates - Turing-complete compile-time metaprogramming via template instantiation.
  • Undefined behavior (UB) - certain constructs (signed overflow, out-of-bounds access) have no defined semantics; the compiler is free to assume they never occur.
  • ABI stability - not guaranteed across compiler versions; major platforms publish stable ABIs only for specific interfaces.

5.0 Standard Library

The C++ Standard Library is partitioned into several functional areas:
  • Containers - vector, array, deque, list, map, unordered_map, set, stack, queue, priority_queue.
  • Algorithms - sort, find, transform, reduce, copy, and ~100 more in <algorithm>. Parallel execution policies since C++17.
  • Iterators - input, output, forward, bidirectional, random-access, and contiguous iterator categories.
  • Ranges - C++20 ranges library: composable, lazy views; std::views::filter, std::views::transform, etc.
  • Concurrency - std::thread, std::mutex, std::atomic, std::future, std::async, std::latch, std::barrier (C++20).
  • Filesystem - std::filesystem (C++17): path manipulation, directory iteration, file status.
  • Utilities - std::optional, std::variant, std::any, std::expected (C++23), std::format (C++20), std::chrono, std::regex.
  • Networking - std::net progressed through technical specifications and is expected in a future standard.

6.0 Interoperability

C++ has strong interoperability with C and, through foreign-function interfaces, with many other languages:
  • C compatibility - C++ is a superset of C (with minor exceptions). C headers can be included directly; the standard library re-exports C headers as <cstdio>, <cstring>, etc.
  • extern "C" linkage - wrapping a declaration in extern "C" { } suppresses C++ name mangling, producing a symbol that C, Python (ctypes/cffi), Rust (FFI), and other languages can call by name.
  • Binary compatibility - C++ objects can be passed across DLL/shared-library boundaries as long as both sides use the same ABI (compiler, calling convention, struct layout).
  • Bindings - pybind11 and nanobind generate Python bindings directly from C++ headers. SWIG supports many target languages. Rust's bindgen auto-generates FFI from C/C++ headers.
