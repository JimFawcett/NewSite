C++ Bites: Object Life Cycle

construction, copy, move, and destruction

About C++ Track

1.0 Synopsis

Every C++ object has a well-defined life cycle: it is constructed, possibly copied or moved, and eventually destroyed. Six special member functions control this cycle. Understanding exactly when each is called - and what the compiler generates when you don't declare them - is foundational to writing correct C++ code.

2.0 Construction

An object comes into existence via a constructor. The compiler selects the matching constructor based on the arguments provided at the point of definition.
  • Default constructor T() - called when no arguments are given. Value-initializes fundamental types to zero when invoked with {}.
  • Parameterized constructors - take arguments that configure initial state.
  • Delegating constructors (C++11) - one constructor calls another of the same class via : ClassName(args) in the member initializer list.
 
struct Point {
  int x, y;
  Point() : x(0), y(0) {}              // default
  Point(int x, int y) : x(x), y(y) {} // parameterized
  explicit Point(int v) : Point(v, v) {}  // delegating
};

Point p1;          // default: (0, 0)
Point p2(3, 4);    // parameterized: (3, 4)
Point p3(7);       // delegating: (7, 7)
Member initialization order follows the order of declaration in the class body, not the order of the initializer list. Initializer lists are preferred over assignment in the constructor body because they run before the body executes and avoid default-constructing then overwriting members.

3.0 Copy Semantics

Copying creates a new object that is an independent duplicate of an existing one.
  • Copy constructor T(const T&) - invoked for direct initialization from another object, pass-by-value, and return-by-value (when copy elision does not apply).
  • Copy assignment T& operator=(const T&) - invoked when assigning to an already-constructed object.
 
class MyString {
public:
  MyString(const char* s)
    : len_(std::strlen(s)), data_(new char[len_ + 1]) {
    std::strcpy(data_, s);
  }
  MyString(const MyString& other)
    : len_(other.len_), data_(new char[len_ + 1]) {
    std::strcpy(data_, other.data_);   // deep copy
  }
  MyString& operator=(const MyString& other) {
    if (this != &other) {
      MyString tmp(other);             // copy-and-swap
      std::swap(data_, tmp.data_);
      std::swap(len_,  tmp.len_);
    }
    return *this;
  }
  ~MyString() { delete[] data_; }
private:
  size_t len_;
  char*  data_;
};

4.0 Move Semantics

Moving transfers ownership of resources from a source object to a destination without copying. The source is left in a valid but unspecified state - it can be destructed safely but should not be used further without reassignment. Mark move operations noexcept so STL containers can use them during reallocation; without noexcept, std::vector falls back to copies to preserve the strong exception guarantee. 
class Buffer {
public:
  Buffer(size_t n) : data_(new int[n]), size_(n) {}
  ~Buffer() { delete[] data_; }

  Buffer(Buffer&& other) noexcept
    : data_(other.data_), size_(other.size_) {
    other.data_ = nullptr;
    other.size_ = 0;
  }
  Buffer& operator=(Buffer&& other) noexcept {
    if (this != &other) {
      delete[] data_;
      data_ = other.data_;  size_ = other.size_;
      other.data_ = nullptr; other.size_ = 0;
    }
    return *this;
  }
private:
  int*   data_;
  size_t size_;
};
std::move(x) is a cast that converts x to an rvalue reference, enabling the compiler to select the move constructor or move assignment operator. It does not move anything by itself.

5.0 Destruction

The destructor ~T() runs when an object's lifetime ends. It releases resources and is the cornerstone of RAII. Destruction happens:
  • automatically when a local variable leaves scope (including during exception unwinding);
  • when delete is called on a heap object;
  • when a container erases or destroys an element;
  • when a member object's containing object is destroyed (members are destroyed in reverse declaration order).
 Base class destructors must be virtual whenever a derived object may be deleted through a pointer to base. Without a virtual destructor, deleting through a base pointer is undefined behavior. 
class Base {
public:
  virtual ~Base() = default;   // virtual: derived destructors run correctly
};

class Derived : public Base {
  std::vector<int> data_;
public:
  Derived() : data_(1000) {}
  ~Derived() { /* data_ freed automatically */ }
};

Base* p = new Derived();
delete p;   // calls ~Derived() then ~Base() - correct
