C++ Bites: Optional and Variant

std::optional, std::variant, and std::any

About C++ Track

1.0 Synopsis

C++ provides three sum-type utilities for expressing "one of several possibilities." std::optional<T> (C++17) holds either a value of type T or nothing. std::variant<Ts...> (C++17) holds exactly one value chosen from a fixed set of types. std::any (C++17) holds any CopyConstructible value with type-erased storage. Headers: <optional>, <variant>, <any>.

2.0 std::optional

std::optional<T> wraps a value that may or may not be present. It stores the value inline - no heap allocation. Accessing the value when it is absent throws std::bad_optional_access (via value()) or returns a default (via value_or()). 
#include <optional>
#include <string>

std::optional<std::string> lookup(int id) {
  if (id == 42) return "found";
  return std::nullopt;
}

auto result = lookup(42);
if (result.has_value())
  std::cout << result.value() << '\n';

std::string s = lookup(0).value_or("not found");
Key operations:
  • has_value() / operator bool - test whether a value is present.
  • value() - return the value; throws std::bad_optional_access if empty.
  • value_or(default) - return the value or the supplied default.
  • operator*, operator-> - unchecked access (undefined behavior if empty).
  • reset() - destroy the contained value, leaving the optional empty.
  • emplace(args...) - construct the value in place.
 C++23 adds monadic operations for composing optionals without nested if chains: and_then(f) calls f if the optional has a value; transform(f) applies f to the value and wraps the result; or_else(f) calls f if the optional is empty.

3.0 std::variant

std::variant<Ts...> is a type-safe union. It always holds exactly one value from its type list and tracks the active type. It cannot be empty (unless std::monostate is included as one of the types). 
#include <variant>
#include <string>

std::variant<int, double, std::string> v = 3.14;

std::visit([](auto&& val) {
  std::cout << val << '\n';
}, v);

if (std::holds_alternative<double>(v))
  std::cout << std::get<double>(v) << '\n';
Key operations:
  • std::get<T>(v) - return the active value if its type is T; throws std::bad_variant_access otherwise.
  • std::get<I>(v) - access by index.
  • std::holds_alternative<T>(v) - test the active type.
  • v.index() - return the zero-based index of the active alternative.
  • std::visit(visitor, v) - call a callable with the active value; the callable must handle all types (exhaustive dispatch).
  • std::monostate - an empty-state sentinel type usable as the first alternative to make a variant default-constructible.
 std::visit with an overloaded lambda set is idiomatic for exhaustive dispatch: 
// Overload helper (C++17)
template<class... Ts> struct overloaded : Ts... { using Ts::operator()...; };
template<class... Ts> overloaded(Ts...) -> overloaded<Ts...>;

std::variant<int, double, std::string> v = "hello";
std::visit(overloaded{
  [](int i)               { std::cout << "int: "    << i << '\n'; },
  [](double d)            { std::cout << "double: " << d << '\n'; },
  [](const std::string& s){ std::cout << "string: " << s << '\n'; }
}, v);

4.0 std::any

std::any stores a single value of any CopyConstructible type. The type is erased at the point of storage; retrieval requires std::any_cast<T> with the exact stored type. For large objects, std::any may heap-allocate; small-buffer optimization is implementation-defined. 
#include <any>
#include <string>

std::any a = 42;
a = std::string("hello");   // replace stored value

try {
  std::string s = std::any_cast<std::string>(a);
  std::cout << s << '\n';
} catch (const std::bad_any_cast&) {
  std::cout << "wrong type\n";
}

if (a.type() == typeid(std::string))
  std::cout << "holds a string\n";

a.reset();   // clear
Use std::any sparingly. It loses compile-time type safety and incurs virtual dispatch on access. Prefer std::variant when the set of possible types is known at compile time.

5.0 Comparison with Pointers

Using a raw pointer as an optional value is a common C idiom but is ambiguous and error-prone:
  • A null pointer may mean "no value," "error," or "uninitialized" - the intent is unclear.
  • The caller may forget to check for null.
  • Heap allocation introduces ownership questions and potential leaks.
 std::optional avoids all three problems: the type itself communicates optionality, has_value() makes the check explicit, and no heap allocation occurs for small types. 
// Pointer: intent unclear, allocation required, null check easy to miss
std::string* find(int id);

// Optional: intent explicit, no allocation, deliberate value access
std::optional<std::string> find(int id);
