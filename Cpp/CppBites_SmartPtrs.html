C++ Bites: Smart Pointers

unique_ptr, shared_ptr, weak_ptr, and custom deleters

About C++ Track

1.0 Synopsis

Smart pointers manage heap-allocated objects through RAII. The standard library provides three:
  • std::unique_ptr - exclusive ownership; destroyed when it goes out of scope.
  • std::shared_ptr - reference-counted shared ownership; destroyed when the last owner releases it.
  • std::weak_ptr - non-owning observer of a shared_ptr; breaks reference cycles.
All three live in <memory> and eliminate the need to call delete manually.

2.0 unique_ptr

std::unique_ptr<T> enforces single ownership. It is non-copyable but movable, so ownership transfers via std::move. Use std::make_unique<T>(args...) to construct - it is exception-safe and avoids a raw new expression. The managed object is destroyed automatically when the unique_ptr goes out of scope. There is zero runtime overhead compared to a raw pointer. Access the underlying pointer with .get(), release ownership without deleting with .release(), and reset to a new pointer with .reset(). 
#include <memory>

// unique ownership
auto u = std::make_unique<int>(42);
std::cout << *u << '\n';   // 42
// auto u2 = u;              // error: not copyable
auto u2 = std::move(u);    // transfer ownership

3.0 shared_ptr

std::shared_ptr<T> allows multiple owners. Each copy increments a reference count stored in a heap-allocated control block. When the count reaches zero the managed object is destroyed. Use std::make_shared<T>(args...) - it allocates the object and control block in a single allocation, which is more cache-friendly and faster than constructing from a raw pointer. Inspect the reference count with .use_count(). The control block adds slight overhead compared to unique_ptr; prefer unique_ptr when shared ownership is not needed. 
// shared ownership
auto s1 = std::make_shared<std::string>("hello");
auto s2 = s1;                       // share ownership
std::cout << s1.use_count() << '\n'; // 2

4.0 weak_ptr

std::weak_ptr<T> holds a non-owning reference to an object managed by a shared_ptr. It does not affect the reference count. To access the object, call .lock(), which returns a temporary shared_ptr - or an empty one if the object has already been destroyed. Use weak_ptr to break ownership cycles in graph-like structures, caches, and observer patterns. 
std::shared_ptr<int> sp = std::make_shared<int>(10);
std::weak_ptr<int>   wp = sp;

if (auto locked = wp.lock()) {      // obtain temporary shared_ptr
  std::cout << *locked << '\n';    // 10
}
sp.reset();                          // destroy object
if (!wp.lock()) std::cout << "expired\n";

5.0 Custom Deleters

Both unique_ptr and shared_ptr accept a custom deleter - a callable invoked instead of delete when the object is released. This makes smart pointers suitable for non-memory resources such as file handles, sockets, and C library handles. 
#include <cstdio>
#include <memory>

// unique_ptr with custom deleter type
auto fileDeleter = [](FILE* f) { if (f) std::fclose(f); };
std::unique_ptr<FILE, decltype(fileDeleter)>
  fp(std::fopen("data.txt", "r"), fileDeleter);

// shared_ptr accepts deleter in constructor
auto sp = std::shared_ptr<FILE>(
  std::fopen("out.txt", "w"),
  [](FILE* f) { if (f) std::fclose(f); }
);

6.0 Common Mistakes

Cyclic shared_ptrs. If object A holds a shared_ptr to B and B holds a shared_ptr back to A, neither is ever destroyed. Break the cycle by making one direction a weak_ptr. Multiple independent shared_ptrs from the same raw pointer. Each creates its own control block; the object is deleted twice. Always copy or move an existing shared_ptr to share ownership. If a class needs to return shared_ptr<this>, inherit from std::enable_shared_from_this<T> and call shared_from_this(). Mixing raw pointers and smart pointers. Storing a raw pointer extracted from a unique_ptr or shared_ptr and then deleting it manually leads to double-free. Use .get() only when the called API requires a raw pointer and the lifetime is clear.
