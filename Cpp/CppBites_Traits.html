C++ Bites: Concepts and Traits

C++20 concepts, type traits, and constrained templates

About C++ Track

1.0 Synopsis

C++20 concepts provide named, readable constraints on template parameters. They replace ad-hoc SFINAE with compiler-enforced requirements and clear error messages. Type traits from <type_traits> are the building blocks: compile-time predicates and type transformations used inside concepts, if constexpr, and template metaprogramming. Together they enable expressive, safe generic code - analogous to Rust's trait bounds, but compile-time only.

2.0 Type Traits

Type traits are struct templates in <type_traits> that expose compile-time information about types. The _v suffix gives a bool value; the _t suffix gives a transformed type.
Trait Meaning
std::is_integral_v<T> true for int, char, long, bool, etc.
std::is_floating_point_v<T> true for float, double, long double
std::is_same_v<T, U> true when T and U are the same type
std::is_copy_constructible_v<T> true when T can be copy-constructed
std::is_trivially_copyable_v<T> true when T can be safely memcpy'd
std::remove_reference_t<T> strips reference from T
std::decay_t<T> strips reference, cv-qualifiers, and array/function decay
std::conditional_t<B, T, F> T if B is true, F otherwise

3.0 Defining a Concept

A concept is defined with the concept keyword followed by a boolean constant expression. The expression may use type traits, requires-expressions, or logical combinations of other concepts. 
#include <concepts>
#include <type_traits>

// define a concept
template<typename T>
concept Numeric = std::integral<T> || std::floating_point<T>;

// constrain a template - three equivalent forms:
template<Numeric T>
T square(T x) { return x * x; }

template<typename T> requires Numeric<T>
T cube(T x) { return x * x * x; }

auto abs_val(Numeric auto x) { return x < 0 ? -x : x; }

4.0 Constraining Templates

The three syntaxes in section 3 are fully equivalent; choose whichever reads best for the context. The requires clause form is most explicit and also supports complex multi-concept conjunctions: requires Numeric<T> && std::copyable<T>. When a concept constraint is violated, the compiler reports which requirement failed and why, rather than the deep substitution errors typical of unconstrained templates. This is the primary practical benefit over pre-C++20 SFINAE.

5.0 Standard Concepts

<concepts> provides a rich set of pre-defined concepts:
Concept Requires
std::integral<T> T is an integer type
std::floating_point<T> T is a floating-point type
std::copyable<T> T is copy-constructible and copy-assignable
std::movable<T> T is move-constructible and move-assignable
std::equality_comparable<T> T supports == and !=
std::totally_ordered<T> T supports all comparison operators
std::invocable<F, Args...> F is callable with Args
std::ranges::range<T> T provides begin() and end()
std::convertible_to<T, U> T is implicitly convertible to U

6.0 requires-expression

A requires-expression tests whether a set of expressions are syntactically valid and, optionally, that they return a type satisfying another concept. It appears inside concept definitions or directly in requires clauses. 
template<typename T>
concept HasSize = requires(T t) {
  { t.size() } -> std::convertible_to<std::size_t>;
};

template<HasSize C>
void printSize(const C& c) {
  std::cout << "size: " << c.size() << '\n';
}
The { expr } -> Concept form checks two things: the expression must compile, and its return type must satisfy the given concept. Without the arrow, only compilability is checked. 
template<typename T>
concept Addable = requires(T a, T b) {
  a + b;                                 // expression must compile
  { a + b } -> std::same_as<T>;        // and return type T
  { a += b } -> std::same_as<T&>;      // and += returns T&
};

7.0 Comparison with Rust Traits

Aspect C++ Concepts Rust Traits
Purpose constrain template arguments define shared behavior; constrain generics
Runtime dispatch no (concepts are compile-time only) yes, via dyn Trait trait objects
Implementation site type must provide the required operations anywhere impl Trait for Type must be explicit
Orphan rules none enforced: impl must be in crate of trait or type
Default methods no direct equivalent in concepts yes, traits can provide default method bodies
Error quality much improved over SFINAE generally excellent
