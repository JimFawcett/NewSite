C++ Bites: Error Handling

exceptions, noexcept, and RAII-based error safety

About C++ Track

1.0 Synopsis

C++ supports error handling through exceptions, error codes, and (C++23) std::expected. The right choice depends on context: exceptions suit truly exceptional conditions where the call site cannot recover locally; error codes suit performance-critical paths or expected failures in tight loops. RAII ties resource cleanup to scope exit, making exception safety automatic when resources are wrapped in owning types.

2.0 Exceptions

The try/throw/catch mechanism unwinds the call stack until a matching handler is found. Catch blocks are checked in order; the first match wins. catch(...) catches any exception. The standard exception hierarchy under std::exception provides std::runtime_error, std::logic_error, std::bad_alloc, and others. Always catch by const reference to avoid slicing. 
#include <stdexcept>
#include <iostream>

void parse(const std::string& s) {
  if (s.empty())
    throw std::invalid_argument("empty input");
}

int main() {
  try {
    parse("");
  } catch (const std::invalid_argument& e) {
    std::cerr << "Error: " << e.what() << '\n';
  } catch (...) {
    std::cerr << "Unknown error\n";
  }
}

3.0 noexcept

Declaring a function noexcept tells the compiler the function will not throw. This enables optimizations — particularly in move semantics, where the STL uses noexcept move constructors to avoid copying on reallocation. If an exception escapes a noexcept function, std::terminate is called immediately. Use noexcept(noexcept(...)) to conditionally propagate the specification. 
// Move constructor must not throw for STL to use it efficiently
class Buffer {
public:
  Buffer(Buffer&& other) noexcept
    : data_(other.data_), size_(other.size_) {
    other.data_ = nullptr;
    other.size_ = 0;
  }
private:
  int*   data_;
  size_t size_;
};

4.0 RAII and Exception Safety

RAII (Resource Acquisition Is Initialization) ensures that resources held by an object are released when the object's destructor runs — including during stack unwinding caused by an exception. Wrapping every resource in an owning RAII type ( std::unique_ptr, std::lock_guard, std::fstream) eliminates the need for try/finally-style cleanup and makes strong exception safety achievable. 
#include <memory>
#include <mutex>
#include <fstream>

void process(const std::string& path) {
  std::ifstream file(path);          // closed in destructor
  std::unique_ptr<int[]> buf(new int[1024]); // freed in destructor

  // if any operation below throws, file and buf are still cleaned up
  doWork(file, buf.get());
}  // destructors run here, even on exception

5.0 Error Codes

std::error_code (from <system_error>) is a portable, type-safe alternative to raw integer error codes. It pairs an integer value with an error_category that provides message strings and comparison semantics. Return an error_code by value for operations that fail frequently and where exception overhead is unacceptable (networking, filesystem hot paths). 
#include <system_error>
#include <filesystem>
#include <iostream>

std::error_code ec;
auto sz = std::filesystem::file_size("data.bin", ec);
if (ec) {
  std::cerr << "file_size failed: " << ec.message() << '\n';
} else {
  std::cout << sz << " bytes\n";
}

6.0 std::expected (C++23)

std::expected<T, E> represents either a value of type T or an error of type E — a type-safe, composable alternative to returning both an error code and an out-parameter. It avoids the overhead of exceptions while keeping error paths explicit and hard to ignore. Use .value(), .error(), and the monadic interface (.and_then(), .or_else(), .transform()) to compose operations. 
#include <expected>
#include <string>

std::expected<int, std::string> parse_int(const std::string& s) {
  try {
    return std::stoi(s);
  } catch (...) {
    return std::unexpected("not an integer: " + s);
  }
}

auto result = parse_int("42");
if (result)
  use(*result);
else
  std::cerr << result.error() << '\n';
