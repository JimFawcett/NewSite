C++ Bites: Idioms

RAII, copy-swap, PIMPL, NVI, CRTP, and rule of zero/three/five

About C++ Track

1.0 Synopsis

C++ idioms are proven patterns that exploit language features to write safer, more maintainable code. They are not design patterns in the Gang-of-Four sense - they are language-level conventions that encode decades of practical experience. Mastering them separates code that compiles from code that behaves correctly under every condition.

2.0 RAII

Resource Acquisition Is Initialization. Resources are acquired in a constructor and released in the destructor. Because the C++ runtime calls the destructor automatically when an object leaves scope - including during stack unwinding from a thrown exception - no explicit cleanup code is needed at call sites. 
class FileHandle {
public:
  explicit FileHandle(const char* path)
    : file_(std::fopen(path, "r")) {
    if (!file_) throw std::runtime_error("cannot open file");
  }
  ~FileHandle() { std::fclose(file_); }   // guaranteed cleanup

  FileHandle(const FileHandle&) = delete;
  FileHandle& operator=(const FileHandle&) = delete;
private:
  FILE* file_;
};
Copy is deleted because two FileHandle objects should not own the same FILE*. The standard library codifies this pattern in std::unique_ptr, std::lock_guard, std::fstream, and others.

3.0 Rule of Zero / Three / Five

These rules govern when to declare the six special member functions: default constructor, destructor, copy constructor, copy assignment, move constructor, and move assignment. Rule of Zero - prefer this. Store resources in member objects that manage themselves (std::string, std::vector, std::unique_ptr). Declare none of the five; the compiler generates correct implementations for all of them. 
// Rule of Zero: no special members needed
struct Config {
  std::string name;
  std::vector<int> values;
  std::unique_ptr<Widget> widget;
};
Rule of Three - if a class manages a raw resource (raw pointer, file handle, mutex), define destructor, copy constructor, and copy assignment. Without explicit copies, the compiler generates a shallow copy that causes double-free on destruction. Rule of Five - extend Rule of Three with move constructor and move assignment. Without them, the compiler suppresses move generation when you declare any of the three, leaving the class copy-only and potentially slow. 
// Rule of Five: raw pointer member requires all five
class Buffer {
public:
  Buffer(size_t n) : data_(new int[n]), size_(n) {}
  ~Buffer()                              { delete[] data_; }
  Buffer(const Buffer& o)               : data_(new int[o.size_]), size_(o.size_)
                                          { std::copy(o.data_, o.data_ + o.size_, data_); }
  Buffer& operator=(const Buffer& o)    { Buffer tmp(o); swap(tmp); return *this; }
  Buffer(Buffer&& o) noexcept           : data_(o.data_), size_(o.size_)
                                          { o.data_ = nullptr; o.size_ = 0; }
  Buffer& operator=(Buffer&& o) noexcept { swap(o); return *this; }

  void swap(Buffer& o) noexcept {
    std::swap(data_, o.data_);
    std::swap(size_, o.size_);
  }
private:
  int*   data_;
  size_t size_;
};

4.0 Copy-and-Swap

Implement operator= by constructing a local copy of the right-hand side, then swapping it with *this. This provides the strong exception guarantee: if the copy constructor throws, the original object is unmodified. 
Buffer& Buffer::operator=(const Buffer& other) {
  Buffer tmp(other);      // copy: may throw, but *this is still intact
  swap(tmp);              // noexcept swap
  return *this;           // tmp (old data) destroyed here
}

// move assignment using the same swap
Buffer& Buffer::operator=(Buffer&& other) noexcept {
  swap(other);
  return *this;
}

5.0 PIMPL

Pointer to Implementation hides private members behind a forward-declared struct pointer. Callers include the header and see only the public interface; the implementation struct is defined only in the translation unit. This reduces compile-time coupling and provides ABI stability - adding private members does not change the class size as seen by consumers. 
// widget.h
class Widget {
public:
  Widget();
  ~Widget();
  void render();
private:
  struct Impl;
  std::unique_ptr<Impl> pImpl_;
};

// widget.cpp
struct Widget::Impl {
  int x, y;
  std::string label;
  void draw() { /* ... */ }
};

Widget::Widget()  : pImpl_(std::make_unique<Impl>()) {}
Widget::~Widget() = default;   // defined here so Impl is complete
void Widget::render() { pImpl_->draw(); }

6.0 NVI - Non-Virtual Interface

Declare public methods as non-virtual; have them delegate to private (or protected) virtual methods. This separates the stable public interface from the customization point, allows the base class to add pre/post conditions around the virtual call, and prevents accidental interface breakage by derived classes. 
class Shape {
public:
  double area() const {          // public, non-virtual
    return doArea();             // delegates to virtual hook
  }
private:
  virtual double doArea() const = 0;  // customization point
};

class Circle : public Shape {
  double radius_;
  double doArea() const override { return 3.14159 * radius_ * radius_; }
public:
  explicit Circle(double r) : radius_(r) {}
};

7.0 CRTP - Curiously Recurring Template Pattern

A base class template parameterized on its own derived class. This enables the base to call derived methods without a vtable - static polymorphism with zero runtime overhead. 
template<class Derived>
class Printable {
public:
  void print() const {
    // Downcast is safe: Derived is always the concrete type
    static_cast<const Derived*>(this)->printImpl();
  }
};

class Point : public Printable<Point> {
public:
  void printImpl() const {
    std::cout << '(' << x_ << ',' << y_ << ")\n";
  }
private:
  int x_ = 0, y_ = 0;
};

Point p;
p.print();   // no virtual dispatch
CRTP is also used for mixin behavior (adding operator overloads or comparison operators to derived classes) and for policy-based design. C++23 "deducing this" offers an alternative for some CRTP use cases without the inheritance overhead.
