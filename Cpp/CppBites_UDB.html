C++ Bites: Undefined Behavior

causes, consequences, detection, and avoidance

About C++ Track

1.0 Synopsis

Undefined behavior (UB) means the C++ standard places no requirements on what the program does when that behavior occurs. The compiler is free to assume UB never happens and to optimize accordingly - producing results that are correct in debug builds and silently wrong in optimized release builds. Understanding UB is essential for writing correct, portable C++.

2.0 What Makes UB Dangerous

The optimizer treats "this code path cannot exhibit UB" as an axiom. When code does contain UB, the optimizer may legally delete branches, reorder operations, or transform the code in ways the programmer did not intend - without any warning. The program may appear correct under a debug build (less optimization, less aggressive assumptions) and break only under -O2 or -O3. Common categories: signed integer overflow, null/dangling pointer dereference, out-of-bounds array access, uninitialized variable reads, and strict aliasing violations. Each is covered in its own section below.

3.0 Signed Integer Overflow

Signed integer overflow is UB. Unsigned overflow is well-defined: it wraps modulo 2N. The optimizer exploits the "no signed overflow" axiom to eliminate checks that only fire after overflow - including security-critical range checks. 
// UB: signed overflow - optimizer may remove the check
int add_signed(int a, int b) {
  if (a + b < a) return INT_MAX;  // optimizer may delete this!
  return a + b;
}

// Safe: use unsigned or check before overflow
bool will_overflow(int a, int b) {
  return b > 0 && a > INT_MAX - b;
}
Safe alternatives: check before the operation (as above), use unsigned when wraparound is acceptable, use __builtin_add_overflow (GCC/Clang), or use std::add_overflow from upcoming C++26 proposals.

4.0 Null and Dangling Pointers

Dereferencing a null pointer is UB. The optimizer may assume that any pointer that is dereferenced is non-null, enabling it to hoist loads above null checks. Dereferencing a dangling pointer (one pointing to freed or out-of-scope memory) is also UB and can corrupt memory or execute arbitrary code. Avoidance: use smart pointers (unique_ptr, shared_ptr) to manage lifetime, prefer references over pointers when nullability is not needed, and use ASAN to detect dangling pointer use at runtime. 
int* p = nullptr;
// *p = 42;   // UB: null dereference

std::vector<int> v{1, 2, 3};
int* dangling = v.data();
v.push_back(4);      // may reallocate; dangling is now invalid
// *dangling = 9;    // UB: dangling pointer

5.0 Out-of-Bounds Access

Indexing a std::vector or C array beyond its bounds with operator[] is UB. It may read or write adjacent memory, corrupt the heap, crash, or silently produce wrong results. Use v.at(i) in debug code - it throws std::out_of_range instead. Use std::span (C++20) to pass array views with known bounds to functions. 
std::vector<int> v{10, 20, 30};
// v[5] = 99;             // UB: out of bounds

v.at(5);                 // throws std::out_of_range in debug

// std::span: bounds-safe array view
#include <span>
void process(std::span<int> s) {
  for (auto x : s) std::cout << x << ' ';
}

6.0 Uninitialized Variables

Reading an uninitialized variable is UB. On some architectures and compilers, the value may be whatever was previously in that memory location; on others, the compiler may use the "never UB" axiom to produce output that changes between builds. Always initialize variables at declaration. The compiler flag -Wuninitialized (included in -Wall) catches many cases. MSAN (-fsanitize=memory, Clang only) detects uninitialized reads at runtime. 
int x;          // uninitialized
// std::cout << x;  // UB: indeterminate value

int y = 0;      // safe: always initialize

// range: prefer value-initialization
std::vector<int> counts(10);      // zero-initialized to {0,0,...,0}
int arr[10] = {};                  // zero-initialized

7.0 Strict Aliasing

The strict aliasing rule says that objects of different types occupy distinct memory and may not alias (with exceptions: char*, unsigned char*, and std::byte* may alias any type). Accessing an object via a pointer of an incompatible type is UB. The optimizer uses this assumption to avoid reloading values from memory after a store through an apparently unrelated pointer. 
// UB: aliasing float as int
float f = 1.0f;
int*  p = reinterpret_cast<int*>(&f);  // strict aliasing violation
int   i = *p;

// Safe C++20: bit_cast preserves bit pattern legally
#include <bit>
int bits = std::bit_cast<int>(f);  // well-defined
std::bit_cast<To>(from) (C++20) reinterprets the bit pattern of from as type To without triggering aliasing UB. Both types must be the same size and trivially copyable. Pre-C++20, use memcpy into a properly typed variable.

8.0 Detection

Tool How to Enable What It Finds
UBSAN -fsanitize=undefined (GCC/Clang) signed overflow, null deref, invalid shift, bad cast
ASAN -fsanitize=address (GCC/Clang), /fsanitize=address (MSVC) out-of-bounds, use-after-free, double-free
MSAN -fsanitize=memory (Clang only) uninitialized reads
Valgrind valgrind --tool=memcheck ./myapp memory errors, leaks; slower but no recompile needed
clang-tidy clang-tidy src.cpp static patterns: null deref, uninitialized, unsafe casts
Run sanitizers in CI on debug builds (-O1 -g is a good balance: some optimization to surface optimizer-dependent UB, with debug info for readable reports). Do not ship sanitizer builds to end users.
