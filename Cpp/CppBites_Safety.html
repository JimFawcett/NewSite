C++ Bites: Safety

memory safety, undefined behavior, sanitizers, and protective patterns

About C++ Track

1.0 Synopsis

C++ does not prevent incorrect memory use at the language level, but a combination of RAII, smart pointers, modern idioms, and runtime tooling (sanitizers) dramatically reduces safety issues in practice. No single technique is sufficient on its own; a defense-in-depth approach - safe coding patterns backed by automated analysis and testing - is the standard professional approach.

2.0 Common Pitfalls

Each item below is undefined behavior (UB): the standard imposes no requirement on what the program does, and the compiler may assume these conditions never occur, enabling optimizations that produce surprising results.
  • Dangling pointer - a pointer that still holds the address of an object after that object has been destroyed. Accessing a dangling pointer is UB.
  • Use-after-free - accessing heap memory after it has been returned to the allocator via delete or free. The allocator may have already reused or returned that memory to the OS.
  • Buffer overflow - writing past the end of an array or allocation. Overwrites adjacent memory, corrupts the stack or heap, and enables exploits.
  • Null pointer dereference - dereferencing a pointer whose value is nullptr. Always check pointers obtained from fallible operations.
  • Double-free - calling delete on the same pointer twice. Corrupts allocator metadata and is a common source of exploitable vulnerabilities.
  • Uninitialized read - reading a variable before it has been assigned a value. Always initialize variables at the point of declaration.

3.0 RAII and Smart Pointers

Replacing raw owning pointers with std::unique_ptr and std::shared_ptr eliminates most memory-management errors at the source. The destructor runs automatically; double-free and use-after-free cannot occur through the smart pointer itself. 
// Unsafe: manual new/delete
int* p = new int[100];
// ... if exception thrown here, p leaks
delete[] p;

// Safe: RAII
auto p = std::make_unique<int[]>(100);
// automatically freed when p goes out of scope
Ownership model:
  • std::unique_ptr<T> - sole ownership. Non-copyable; transfer via std::move. Zero overhead compared to a raw pointer.
  • std::shared_ptr<T> - shared ownership via reference counting. Incurs a control block allocation and atomic reference count updates.
  • std::weak_ptr<T> - non-owning observer of a shared_ptr-managed object. Lock to obtain a temporary shared_ptr; safe for breaking ownership cycles.
The C++ Core Guidelines rule: raw pointers and references are never owning. If a raw pointer appears in an interface, it should always be documented as non-owning.

4.0 Bounds Safety

Array subscript operators do not perform bounds checking by default. Out-of-bounds access is UB. 
std::vector<int> v = {1, 2, 3};

v[5];       // undefined behavior: no bounds check
v.at(5);    // throws std::out_of_range: bounds check

// std::span (C++20): safe view of contiguous data
void process(std::span<int> data) {
  for (int x : data) std::cout << x << ' ';
}
process(v);  // no copy, bounds preserved in span
Debug-build options that add instrumentation:
  • GCC/libstdc++: define _GLIBCXX_DEBUG before including STL headers.
  • MSVC: checked iterators are enabled in debug builds by default via _ITERATOR_DEBUG_LEVEL.
  • libc++ (Clang): -D_LIBCPP_ENABLE_ASSERTIONS=1 enables iterator and container checks.

5.0 Sanitizers

Sanitizers instrument the compiled binary to detect errors at runtime. They impose a build-time cost (larger binary, slower execution) but catch bugs that static analysis misses.
SanitizerWhat it catchesCompiler flag
AddressSanitizer (ASAN) Heap/stack buffer overflows, use-after-free, use-after-scope, double-free -fsanitize=address (GCC/Clang)
/fsanitize=address (MSVC)
UndefinedBehaviorSanitizer (UBSAN) Signed integer overflow, null pointer dereference, misaligned access, invalid enum values -fsanitize=undefined
LeakSanitizer (LSAN) Memory leaks at program exit -fsanitize=leak (included in ASAN on Linux)
ThreadSanitizer (TSAN) Data races between threads -fsanitize=thread
Combine ASAN and UBSAN in a single build for broad coverage: 
// Compile with:
// g++ -fsanitize=address,undefined -g -O1 myfile.cpp

// ASAN output on use-after-free:
// ==ERROR: AddressSanitizer: heap-use-after-free
//  READ of size 4 at 0x... thread T0
//  #0 main myfile.cpp:12

6.0 Static Analysis

Static analyzers examine the source code without running it, catching a different class of issues than sanitizers.
  • clang-tidy - a Clang-based linter with hundreds of checks. The cppcoreguidelines-* and modernize-* check sets enforce the C++ Core Guidelines and flag obsolete patterns. Integrate with compile_commands.json or CMake.
  • cppcheck - focused on detecting undefined behavior, memory leaks, and portability issues. Fast, low false-positive rate.
  • MSVC /analyze - Microsoft's static analysis checker, integrated into the compiler. Runs the SAL (Source Annotation Language) checks and Core Guidelines checker.
  • PVS-Studio, Coverity, SonarQube - commercial tools with broader interprocedural analysis.
No tool finds everything. A CI pipeline that runs at minimum clang-tidy and ASAN+UBSAN on every pull request catches the vast majority of common safety issues before code ships.
