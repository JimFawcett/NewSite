C++ Bites: Synchronization

mutex, lock_guard, condition_variable, and atomic

About C++ Track

1.0 Synopsis

When multiple threads share mutable data, synchronization prevents data races - which are undefined behavior in C++. The standard library provides three complementary mechanisms:
  • <mutex> - mutual exclusion locks and RAII wrappers.
  • <condition_variable> - blocking coordination between producer and consumer threads.
  • <atomic> - lock-free operations on scalar types.

2.0 std::mutex

std::mutex provides exclusive ownership: only one thread holds the lock at a time. Calling lock() on an already-locked mutex blocks until the owner calls unlock(). Do not call lock and unlock directly - an exception between them leaks the lock. Always use an RAII wrapper. Other mutex variants: std::recursive_mutex (same thread may lock multiple times), std::timed_mutex (try_lock_for / try_lock_until), std::shared_mutex (reader-writer lock, C++17).

3.0 lock_guard and unique_lock

std::lock_guard<std::mutex> locks on construction and unlocks on destruction. It is non-movable and non-copyable, so it works only within a single scope. Prefer it when you need a simple scoped lock. std::unique_lock<std::mutex> is more flexible: it supports deferred locking (std::defer_lock), timed locking, manual unlock/relock, and move semantics. std::condition_variable::wait requires a unique_lock. 
#include <mutex>
#include <vector>

std::mutex mtx;
std::vector<int> shared_data;

void producer(int v) {
  std::lock_guard<std::mutex> lock(mtx);  // RAII lock
  shared_data.push_back(v);
}  // lock released here
C++17 introduced std::scoped_lock for locking multiple mutexes simultaneously without deadlock risk. Prefer it over lock_guard when multi-mutex locking is needed.

4.0 condition_variable

std::condition_variable lets a thread block until a condition becomes true, without busy-waiting. The producer signals with notify_one() or notify_all(). The consumer waits with wait(lock, predicate). The predicate form of wait is essential. Spurious wakeups - where the thread wakes without a notification - can occur on any platform. The predicate re-checks the condition and goes back to sleep if it is still false. 
#include <mutex>
#include <condition_variable>
#include <queue>

std::mutex              mtx;
std::condition_variable cv;
std::queue<int>         q;

void producer(int v) {
  {
    std::lock_guard<std::mutex> lk(mtx);
    q.push(v);
  }
  cv.notify_one();
}

void consumer() {
  std::unique_lock<std::mutex> lk(mtx);
  cv.wait(lk, [&]{ return !q.empty(); });  // handles spurious wakeup
  int v = q.front(); q.pop();
  lk.unlock();
  // process v ...
}

5.0 std::atomic

For simple shared counters or flags, std::atomic<T> avoids a mutex entirely. Operations such as ++, --, load(), store(), and compare_exchange_strong() are guaranteed to be indivisible. std::atomic works for trivially copyable types; practical hardware support is best for integral types and pointers. The default memory order (std::memory_order_seq_cst) is the safest and the slowest. For performance-critical code, weaker orders such as memory_order_relaxed or memory_order_acquire/memory_order_release can be used with care. 
#include <atomic>

std::atomic<int> counter{0};

void increment() {
  ++counter;    // atomic: no data race
}

std::atomic<bool> done{false};

6.0 Deadlock Prevention

Deadlock occurs when two threads each hold a lock the other needs. The root fix is a consistent lock ordering: always acquire locks in the same global order across all threads. When consistent ordering is impractical, use std::scoped_lock (C++17), which locks multiple mutexes simultaneously using a deadlock-avoidance algorithm. 
#include <mutex>

std::mutex m1, m2;

void safeTransfer() {
  // C++17: locks both atomically, no deadlock
  std::scoped_lock lock(m1, m2);
  // ... operate on resources guarded by m1 and m2
}

// C++11/14 equivalent using std::lock + adopt_lock
void safeTransferOld() {
  std::lock(m1, m2);                              // lock both atomically
  std::lock_guard<std::mutex> lg1(m1, std::adopt_lock);
  std::lock_guard<std::mutex> lg2(m2, std::adopt_lock);
  // ...
}
