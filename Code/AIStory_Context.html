AI Story: Context

context levels, instruction files, commands, settings, output steering

12.  Context

Claude Code reads context from multiple sources before responding to any request. Understanding which files it reads, what it writes, how to control scope, and how to phrase requests lets you get precise, reliable output without repeated correction. This chapter covers the full context and configuration system.

12.1  Context Levels

Claude Code loads instructions from up to four levels, applied in order of increasing specificity. More specific always wins.
  • User (global) - ~/.claude/CLAUDE.md. Loaded once at startup, applies to every project. Put response style, code conventions, and git rules here - anything that should be consistent across all work.
  • Project - <repo-root>/CLAUDE.md. Loaded when Claude Code opens a git repository. Covers architecture, tech stack, build commands, key conventions, and entry points for that codebase. Applies to all files unless a subfolder overrides it.
  • Folder - <subdirectory>/CLAUDE.md. Loaded when work enters that directory. Narrows or overrides the project-level instructions for a subtree - useful when a subproject has distinct conventions (for example, a Python agent directory with different run commands or style rules).
  • File - active file contents or IDE selection. Read per-request. The most specific context; drives the immediate task.
Precedence: file > folder > project > user. Memory (~/.claude/projects/<slug>/memory/) is separate from CLAUDE.md. It persists facts Claude learns across conversations. settings.json is separate still - it configures harness behavior (permissions, plugins), not instruction context.

12.2  Local Files Claude Writes

All context management is local. Your prompt and code are sent to Anthropic servers for inference, but settings, memory, and CLAUDE.md never leave your machine.
FileWho writes itWhere
settings.json User / harness ~/.claude/ or .claude/
CLAUDE.md You ~/.claude/, repo root, or subdirs
CLAUDE.local.md You Project root (gitignored, personal)
MEMORY.md Claude ~/.claude/projects/<slug>/memory/
.claude.json Harness ~/ (OAuth, MCP configs, caches)
MEMORY.md is written by Claude during sessions as it learns preferences, not by the harness. Only the first 200 lines load at session start; Claude creates additional topic files in the same folder as needed.

12.3  Instruction Files

Only CLAUDE.md is formally recognized - loaded automatically at startup. Everything else is user-created convention, read on request rather than automatically.
  • CLAUDE.md - the one file Claude Code reads at startup. Put build commands, architecture notes, and conventions here.
  • CLAUDE.local.md - project-local personal notes (gitignored); for things you don’t want committed.
  • AGENTS.md - if your repo already has this for other AI tools, reference it inside CLAUDE.md with @AGENTS.md so both tools share the same instructions.
  • Constitution.md - a user convention (not recognized by Claude Code) for capturing values, constraints, or principles; ask Claude to read it explicitly when needed.
  • Spec.md, CONVENTIONS.md, Notes.md - plain docs Claude won’t load automatically but can read on request.
The practical pattern: put anything you want Claude to always know in CLAUDE.md. Put reference material Claude should read on demand in separate files and ask it to read them when relevant. The three CLAUDE.md levels map directly to content scope:
  • User ~/.claude/CLAUDE.md - universal content: response style, code conventions, git rules that apply everywhere.
  • Project <repo-root>/CLAUDE.md - project content: architecture, tech stack, build commands, entry points.
  • Folder <subdir>/CLAUDE.md - folder content: overrides for a subtree with distinct conventions.

12.4  Controlling What Claude Sees

Claude accesses your full project directory, git state, terminal environment, and CLAUDE.md instructions. Control scope with:
  • @filename - mention a file to add it to context.
  • /add-dir - give Claude permanent access to additional directories.
  • IDE selection - highlight code and Claude reads it.
  • CLAUDE.md - persistent instructions load at every session start (first 25 KB / 200 lines).
  • Auto memory - Claude saves learnings to ~/.claude/projects/.../MEMORY.md, loaded each session.
  • /context - shows what is consuming context window space.
  • /compact - summarizes conversation to free context when it fills.
 Use plan mode (Shift+Tab) for read-only context gathering before edits.

12.5  Models and Effort

Switch models during a session with /model (opens a picker) or /model <name> (direct). At startup use claude --model opus, or set ANTHROPIC_MODEL=opus as an environment variable, or add "model": "opus" to ~/.claude/settings.json. Available aliases (resolve to latest versions):
  • sonnet, opus, haiku - latest stable releases.
  • opus[1m], sonnet[1m] - 1M token context windows.
  • Default varies by plan: Opus on Max/Team/Enterprise, Sonnet on Pro.
 Effort levels control reasoning depth. Set with /effort or the environment variable CLAUDE_CODE_EFFORT_LEVEL. Levels: low, medium, high (default), xhigh, max.

12.6  Customization

Settings files use a hierarchical priority (lowest to highest):
  1. User global: ~/.claude/settings.json
  2. Team/project: .claude/settings.json (committed)
  3. Local project: .claude/settings.local.json (gitignored)
  4. Managed/MDM policies (enforced by IT)
 Key customization points:
  • /config - interactive UI for settings.
  • /keybindings - edit ~/.claude/keybindings.json to rebind keys.
  • /theme - pick light/dark or create custom CSS overrides.
  • /memory - refine CLAUDE.md instructions.
  • hooks in settings - lifecycle scripts (pre/post command, on error, on complete).
  • permissions - allow/ask/deny lists for tools and commands.

12.8  Settings Structure

A representative settings.json: 
{
  "permissions": {
    "allow": ["Bash(npm run *)", "Read(./src/**)"],
    "ask":   ["Bash(git push)"],
    "deny":  ["Bash(rm -rf *)", "Read(./.env)"]
  },
  "env":   { "NODE_ENV": "development" },
  "model": "opus",
  "hooks": {
    "onPreCommand": "echo 'Starting...'",
    "onComplete":   "echo 'Done'"
  },
  "autoMemoryEnabled": true,
  "editorMode": "vim"
}
Scope precedence (highest to lowest): managed policies, CLI flags, local project, team/project, user global. Permissions merge across scopes rather than override - managed policies can tighten but not loosen lower-level rules. Use glob patterns for specificity: Bash(curl *), Read(~/.env). Changes to permissions, hooks, env, and theme reload immediately. Changes to model and outputStyle require a restart.

12.9  Phrases to Steer Output

Prompt phrasing directly shapes the form and quality of Claude’s response. Specificity beats vagueness: "Return only the function signature, no body, no explanation" outperforms "be brief." Combining a scope phrase with a format phrase gives the tightest control. Reasoning depth
  • "Think carefully before answering"
  • "Think step by step" / "Work through this methodically"
  • "Reason through this out loud"
  • "Show your reasoning"
 Confidence and speculation
  • "Do not speculate - return only what you know"
  • "If you’re uncertain, say so explicitly"
  • "Flag any assumptions you’re making"
  • "If you don’t know, say ‘I don’t know’ rather than guessing"
  • "Return facts, not opinions"
  • "Do not infer - report only what is stated"
 Scope and completeness
  • "Be exhaustive" / "Cover all cases"
  • "Answer only the question asked - nothing more"
  • "Do not omit edge cases"
  • "Include everything relevant, leave out everything that isn’t"
 Format and length
  • "Be concise" / "Be brief" / "In one sentence"
  • "Use bullet points" / "Use a table" / "Use numbered steps"
  • "No preamble - start with the answer"
  • "No trailing summary"
  • "Return only the code, no explanation"
  • "Follow this format exactly: ..."
 Directness and hedging
  • "Give a single recommendation, not options"
  • "No caveats unless they materially change the answer"
  • "Don’t hedge - commit to an answer"
  • "Skip the pleasantries"
 Tone and audience
  • "Write for an expert audience" / "Write for a beginner"
  • "Use active voice"
  • "No jargon" / "Use technical terminology"
  • "Write in my voice" (after showing examples)
 Role and perspective
  • "Act as a senior software engineer"
  • "You are an expert in X"
  • "From the perspective of a security auditor..."
  • "As a skeptic, identify weaknesses in..."
 Verification
  • "Double-check your answer before responding"
  • "Check for edge cases"
  • "Are there any counterexamples?"
  • "What might be wrong with that answer?"
  • "Play devil’s advocate"
 Constraint on approach
  • "Without using X"
  • "Using only standard library"
  • "Without modifying Y"
  • "Preserve existing behavior"
