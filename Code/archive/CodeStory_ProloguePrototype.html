0.  Synopsis

Code Story documents a progression through AI-assisted development workflows - from interactive chat sessions through CLI agents, direct API calls, and tool-using pipelines, arriving at spec-driven generation where formal documents replace ad hoc prompts. Two reference projects, TextFinder and PageValidator, run as concrete targets throughout: both are implemented in five languages, built from specs, and measured so claims about performance and correctness rest on numbers rather than intuition. The main story sequence (chapters 1–4) covers the Code Track layout, a scratch experiment arena, and the two reference projects in detail. A parallel set of technique chapters covers the AI tooling that makes that work possible.

0.1  Recurring Themes

Four ideas cut across every chapter:
  1. Spec-driven generation - Constitution.md constrains values, Structure.md defines layout and dependency rules, Spec.md contracts each public item. AI implements one file at a time against those documents; violations are rejectable by inspection, and the same spec drives implementations in multiple languages.
  2. Cross-language comparison - TextFinder and PageValidator share an identical CLI and produce identical output in Rust, C++, C#, and Python. Performance differences are measured, not guessed, and the causes - regex engine choice, runtime startup cost, OS syscall strategy - are explained from first principles.
  3. Layered AI usage - Each technique chapter adds one capability: chat bots → CLI sessions → direct API → tool use → agentic pipelines → skill libraries. Later chapters compose earlier ones; nothing is discarded.
  4. Metrics as feedback - Line counts, scope counts, complexity scores, and wall-clock timing are collected from the start. The arena chapter gives you the tool chain; subsequent chapters use it to validate every claim.

0.2  Chapter Map

Main Sequence

Chapter Key ideas
1 - Code Track Map Directory layout and purpose of every file family in the Code track: root HTML pages (CodeStory, CodeBites, AIStory, RepoCode), top-level directories, and the AI/ subfolder in detail - from demo agents through skill modules to context notes.
2 - Experiment Arena Per-language scratch workspace; metric tools (tokei, scc, radon, cargo clippy, cppcheck); timing tools (hyperfine, cProfile, cargo flamegraph, BenchmarkDotNet); snakeviz and graphviz for visualization; a short session ritual for keeping experiments organized and reproducible.
3 - TextFinder Project Three-component architecture (CommandLine, DirNav, Output) wired by EntryPoint with no cross-library dependencies; identical CLI across five implementations; skip-list strategy for build artifact directories; performance table with explanation (Python leads on I/O-bound workloads because its hot path is C and its regex engine is DFA-based); Rust optimization story showing that DirEntry::file_type() eliminates redundant stat syscalls, accounting for a 33% median reduction.
4 - PageValidator Project HTML structural validator using the same three-component pattern and spec-driven workflow as TextFinder; cross-language comparison with a CPU-bound workload, providing a counterpoint to TextFinder’s I/O-bound results.

Technique Chapters

Chapter Key ideas
Code AI CLI Claude Code and Gemini CLI as terminal coding partners; /init generates a project-specific CLAUDE.md loaded automatically in every subsequent session; multi-step compound tasks (read, analyze, edit, run tests) within a single session; hooks in .claude/settings.json for automating recurring actions such as linting after every edit.
Agentic AI Analyze → plan → generate → test four-call pipeline; carrying structured state between calls using plain Python dicts or JSON files; human-in-the-loop checkpoints after high-risk steps; when autonomy helps and when it does not.
Skills AI Named, reusable tool definitions eliminating copy-paste maintenance; anatomy of a skill (Python function + docstring description + input_schema dict); a code_metrics example; composing multiple skills in a single agent call so the model chooses what to invoke and in what order.
Spec-Driven Development Three-document approach: Constitution.md (one-page hard constraints), Structure.md (package layout and cross-module dependency rules), Spec.md (per-function signatures, preconditions, postconditions, and one example); seven-step workflow from spec to tested, constitution-verified implementation; the same Constitution governs multiple language implementations.

0.3  Setup

The story assumes a working language toolchain and an Anthropic API key for the AI chapters. Minimum requirements by area:
  • Rust - rustup + cargo. Chapters 3–4 use a Cargo workspace; arena experiments use cargo new.
  • C++ - MSVC, GCC, or Clang; CMake 3.28+ for the named-module build used in TextFinder.
  • C# - .NET 10 SDK; dotnet build from the project root.
  • Python - 3.10+; pip install anthropic for AI chapters; arena metrics tools (pip install radon, cargo install hyperfine).
  • Anthropic API key - set as ANTHROPIC_API_KEY; required for the CLI, Agentic AI, Skills AI, and Spec-Driven Development technique chapters and for the generate_part.py scripts in chapters 3–4.
