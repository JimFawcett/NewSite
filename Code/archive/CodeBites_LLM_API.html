1.0 - Introduction

Calling Claude directly from code gives you full programmatic control: you choose the model, the system prompt, the context, the output format, and whether to stream. This chapter covers the Anthropic Python SDK from a minimal completion through structured output, streaming, and prompt caching.
Why write code that calls the API?
  1. You can embed AI calls inside larger programs: scripts, batch processors, analysis pipelines.
  2. You control the system prompt precisely - the AI’s persona, constraints, and output format are not left to the chat interface defaults.
  3. Structured output (JSON) lets you parse and act on AI responses programmatically.
  4. Prompt caching reduces cost and latency when the same large context is reused across many calls.

2.0 - Minimal Completion

The smallest working call: one user message, one response. The client reads ANTHROPIC_API_KEY from the environment automatically. 
import anthropic

client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY from environment

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": "Explain Rust ownership in three sentences."
    }]
)
print(response.content[0].text)
response.content is a list; index zero holds the text block for a plain-text response. For multi-part responses - tool calls, image outputs - iterate the list and dispatch on block.type.

3.0 - Structured Output

Put the JSON schema in the system prompt; the model returns JSON you can parse directly. Validate with pydantic for robustness. 
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    system='Respond only with valid JSON matching {"summary": str, "complexity": int}.',
    messages=[{"role": "user", "content": code_text}]
)
import json
result = json.loads(response.content[0].text)
The system prompt is the contract: if you change the schema there, change the parsing code to match. Keeping schema and parser in the same file makes that alignment easy to verify.

4.0 - Streaming

Use the stream context manager to receive tokens as they are generated. Streaming reduces perceived latency for long responses and lets you display progress without waiting for the full reply. 
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": prompt}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
The context manager closes the connection cleanly when the block exits, even on early break. For pipelines that feed streamed output into a downstream process, accumulate into a StringIO buffer inside the loop rather than printing.

5.0 - Prompt Caching

Mark large, reused context blocks with cache_control to avoid re-processing them on every call. Cached tokens cost roughly 10× less and return faster. Most useful when every call in a session loads the same large file. 
messages=[{
    "role": "user",
    "content": [
        {
            "type": "text",
            "text": large_context,
            "cache_control": {"type": "ephemeral"}
        },
        {"type": "text", "text": "Summarize the above."}
    ]
}]
Cache lifetime is five minutes. If calls are spaced further apart than that, the cache misses and the full token cost applies. For batch jobs, keep calls tightly grouped or accept the miss as expected. Only the content blocks marked with cache_control are cached. The variable follow-up question at the end of the list is sent fresh each time, which is the intended pattern: stable context cached, dynamic question uncached.
