terms and concepts used across all Code track pages

Glossary

Terms are organized by domain. Each entry names the concept and gives a one- or two-sentence definition as used in the Code track pages.

1.  AI and Machine Learning Fundamentals

  • Artificial Intelligence (AI)
    Systems that perform tasks normally associated with human cognition: perception, reasoning, language, and decision-making.
  • Machine Learning (ML)
    A subset of AI where the system learns from data rather than being explicitly programmed. The programmer designs the training loop; the parameters are learned.
  • Deep Learning
    ML using neural networks with many layers. Layers progressively extract higher-level features from raw input.
  • Large Language Model (LLM)
    A deep learning model trained on text at massive scale. “Large” refers to parameter count (billions to trillions) and training data (trillions of tokens).
  • Generative AI
    AI that produces new content (text, images, code, audio) rather than classifying or predicting from fixed categories. LLMs are the dominant form for text and code.
  • Transformer
    Neural network architecture introduced by Google in 2017 that underlies all major LLMs. Its core operation is attention: every token can draw information from every other token in the context.
  • Attention
    The key operation in a transformer. Every token in the input attends to (draws information from) every other token in the context window, making LLMs sensitive to long-range dependencies and prompt structure.
  • Token
    The unit of text an LLM processes. A vocabulary of roughly 100,000 sub-word pieces; common words are one token, rare words split into several.
  • Tokenizer
    The component that converts raw text into token IDs and back. The same tokenizer is used for both input and output.
  • Context Window
    The maximum number of tokens the model can attend to in a single inference call, covering both input prompt and generated output.
  • Knowledge Cutoff
    The date beyond which an LLM has no training data. Information created after this date must be supplied in the prompt.
  • Training
    The phase where the model sees billions of examples and adjusts its parameters to minimise prediction error. Done once (or periodically) by the model provider, not by the API user.
  • Inference
    The phase where the trained model processes a new input and produces output. Every API call is one inference pass.
  • Fine-tuning
    Additional training on a smaller, task-specific dataset after the base model is trained. Some providers expose this as an API feature; Anthropic does not (as of 2026).
  • KV-cache
    Key-value pairs computed during the attention pass that can be reused for identical prompt prefixes, reducing cost and latency on repeated requests.

2.  Claude API and Prompting

  • Messages API
    The primary Anthropic endpoint for calling an LLM from code. Accepts an array of alternating user and assistant messages and returns the next assistant response.
  • Model ID
    The exact identifier string (e.g. claude-sonnet-4-6) that selects a specific model version. Should be pinned in production to prevent unexpected behavior changes.
  • Prompt
    Everything sent to the model before it generates a response, including the system prompt and all prior conversation turns.
  • System Prompt
    A top-level instruction block that sets the model’s behavior, persona, and constraints for the entire conversation. Processed before the first user turn.
  • User Turn
    A message with role “user” containing input from the caller. Tool results are also delivered in user turns, wrapped in a tool_result content block.
  • Assistant Turn
    A message with role “assistant” containing the model’s response. Previous responses are appended to the messages array so the model can see its own prior output.
  • max_tokens
    API parameter capping the number of output tokens the model may produce. Generation stops at this limit even if the response is incomplete.
  • stop_reason
    Field in the API response indicating why the model stopped: end_turn, max_tokens, stop_sequence, or tool_use.
  • Content Block
    A unit of message content. Can be a text block, a tool_use request block, or a tool_result block.
  • Multi-turn Conversation
    A series of alternating user and assistant messages where the full history is sent to the model on each call. Enables coherent follow-up questions.
  • Zero-shot Prompting
    Asking the model directly without supplying examples. Works well for tasks the model has seen many times in training.
  • Few-shot Prompting
    Supplying two to five input/output examples before the real query to teach the model the exact format and tone expected.
  • Chain-of-Thought (CoT)
    Asking the model to reason step by step before giving a final answer. Improves accuracy on multi-step problems.
  • Grounding / RAG
    Supplying relevant facts in the prompt so the model can reason over current or private information it was not trained on. RAG stands for Retrieval-Augmented Generation.
  • Structured Output
    Constraining the model’s output to a predictable format (typically JSON) that can be parsed programmatically. Specified via a JSON Schema in the system prompt or the API parameter.
  • Assistant Prefill
    Starting the assistant turn yourself to force a specific output format, for example beginning with { to guarantee JSON output.

3.  Streaming, Caching, and Reliability

  • Streaming
    Delivering tokens as they are generated via server-sent events rather than waiting for the complete response. Reduces perceived latency for the user.
  • Server-Sent Events (SSE)
    A persistent HTTP connection over which the server pushes a sequence of typed event objects. The mechanism underlying Claude streaming responses.
  • Delta
    A small chunk of text (typically one to five tokens) carried in each streaming event. Accumulate deltas to reconstruct the full response.
  • Prompt Caching
    Reusing the KV-cache for the prefix of a prompt that stays the same across repeated requests. Reduces cost and latency on calls that share a large, stable context.
  • cache_control
    API metadata attached to a content block to mark it as a cache breakpoint. Type ephemeral caches the prefix up to that point for five minutes.
  • Cache Hit / Cache Miss
    A hit occurs when a subsequent request contains the same prefix and the cached KV state is reused. A miss means the prefix must be recomputed at full cost.
  • TTL (Time-To-Live)
    The five-minute expiration window for a prompt cache entry. After TTL the cache expires and the next request pays the full write cost again.
  • Exponential Backoff
    A retry strategy where the delay between attempts doubles on each failure. Prevents overwhelming the API during transient errors.
  • Token Budget
    A maximum cumulative token usage cap set for a session or workflow to prevent runaway costs.
  • RateLimitError
    Exception raised when too many requests per minute or too many tokens per minute are sent to the API. Safe to retry with backoff.

4.  Tool Use and Agents

  • Tool
    A function callable by the model, described by name, description, and a JSON Schema for its input parameters. The model requests tool execution; the caller executes and returns the result.
  • Tool Definition
    The JSON structure passed to the API specifying a tool’s name, description, and input_schema.
  • Tool-use Block
    A structured content block produced by the model when it requests execution of a tool, containing the tool name and input arguments.
  • Tool Result Block
    A tool_result content block in a user turn that delivers the output of an executed tool back to the model.
  • Tool-use Loop
    The three-step exchange: send message with tools defined, model responds with tool_use, caller executes and returns tool_result, model continues.
  • Parallel Tool Use
    The model requesting several tools in a single response. All requested tools can be executed concurrently before returning results.
  • Agent
    An LLM-powered system that takes actions across multiple steps, observes results, and decides the next action until a goal is reached.
  • Agentic Loop
    The fundamental agent pattern: receive task, plan, execute a tool, observe the result, repeat until the task is complete or a step limit is reached.
  • Sub-agent
    A nested agent call with a clean context used to delegate a sub-task, keeping the parent agent’s context from growing unbounded.
  • Step Limit
    A maximum number of agentic loop iterations. When reached the agent fails loudly rather than silently consuming tokens indefinitely.
  • Least-Privilege
    The principle of giving an agent only the tools needed for its current task, reducing the blast radius of mistakes.
  • Human-in-the-Loop
    A design pattern that inserts a human approval step before the agent executes high-risk or irreversible tools.

5.  AI Development Patterns and Skills

  • Spec-Driven Development
    A workflow where a Constitution.md (hard constraints), a Structure.md (component layout), and a Spec.md (signatures and contracts) are written first, then used to drive code generation.
  • Constitution.md
    A project-level file that states invariant constraints the generated code must satisfy. Passed to the model with every generation request.
  • Spec.md
    A component-level file that provides function signatures, type contracts, and behavioral requirements used to generate or regenerate that component.
  • generate_part.py
    A per-project script that calls the Claude API to regenerate a named component from its Spec.md, using Constitution.md and Structure.md as context.
  • Skill
    A reusable, named agent capability packaged with its tool definitions, prompt instructions, and metadata. Can be loaded into an agent on demand.
  • Decomposition Pattern
    Splitting a task into ordered subtasks, presenting them to the model one at a time to avoid overwhelming the context.
  • Planning Pattern
    Having the model propose a full approach and pausing for human approval before proceeding to implementation.
  • Scaffolding Pattern
    Providing interface skeletons (signatures, struct layouts, empty bodies) and asking the model to fill in the implementation.
  • ReAct Pattern
    Interleaving reasoning steps with actual tool calls so the model observes real results before deciding the next action.
  • Self-Critique Pattern
    The model produces an answer, critiques it against stated criteria, then revises based on its own critique.
  • Test-Driven Prompting
    Writing tests or acceptance criteria first, then asking the model for an implementation that satisfies them.
  • Role Framing
    Assigning an expert persona in the system prompt to bias the model’s vocabulary, rigor, and priorities toward a specific domain.

6.  Code Projects

  • TextFinder
    A tool that walks a directory tree and reports every file whose content matches a user-supplied regular expression. Implemented in Rust (baseline and optimized), C++, C#, and Python with a shared three-component architecture and CLI.
  • PageValidator
    A tool that checks HTML files for structural correctness against eight rules. Implemented in Rust, C++, C#, and Python using a shared four-component pipeline architecture.
  • CommCompare
    A project comparing socket-based message-passing communicators built in Rust and C++, evaluating performance, complexity, code size, and safety side by side.
  • Three-Component Architecture
    The TextFinder design pattern: CommandLine, DirNav, and Output are independent libraries wired together by an entry point with no cross-library dependencies.
  • Pipeline Architecture
    The PageValidator design pattern: a strictly linear chain Tokenizer → Lexer → Validator → EntryPoint where each stage feeds the next.
  • CommandLine Component
    Parses /Key [Value] tokens from argv and exposes typed option values to the entry point.
  • DirNav Component
    Performs a depth-first directory walk and fires callbacks on each directory and file entered.
  • Tokenizer Component
    Reads raw HTML and emits a flat token stream with no knowledge of HTML grammar or nesting structure.
  • Lexer Component
    Groups raw tokens from the Tokenizer into structured lexemes with source positions for use by the Validator.
  • Validator Component
    Drives the Lexer, maintains an open-tag stack, applies the eight validation rules, and returns a complete error report.

7.  Development Tools

  • VS Code
    Lightweight, extensible editor from Microsoft that runs on Windows, macOS, and Linux. The primary editor used across all Code track projects.
  • CMake
    Cross-platform build-system generator. Reads CMakeLists.txt and generates platform-native build files (Makefiles, Visual Studio projects, Ninja files).
  • Clang / clangd
    Clang is the C/C++ compiler front-end; clangd is the language server that powers VS Code C++ extension features including diagnostics, completions, and formatting.
  • rust-analyzer
    The Rust language server providing type inference, borrow-checker feedback, and inlay hints inside VS Code.
  • dotnet CLI
    The command-line interface for .NET. Core commands: dotnet new, dotnet build, dotnet run, dotnet test.
  • cargo
    The Rust package manager and build tool. Manages dependencies, compiles crates, runs tests, and publishes to crates.io.
  • hyperfine
    A cross-platform command-line benchmarker. Runs a command N times with cache warm-up and reports mean, standard deviation, and outliers.
  • tokei
    A fast, accurate code line counter. Reports lines of code, comments, and blanks broken down by language and file.
  • scc (Sloc Cloc and Code)
    Like tokei but adds estimated cyclomatic complexity and cost-to-develop columns. Useful for comparing equivalent programs across languages.
  • radon
    Python static analysis tool reporting cyclomatic complexity and maintainability index per function. Install: pip install radon.
  • code_metrics.py
    A Code track utility script reporting line counts, function counts, and blank/comment ratios per source file across a project directory.
  • CodeWebifier
    A .NET console tool that converts source files to syntax-highlighted HTML fragments suitable for embedding in the site. Produces one HTML file per source file.
