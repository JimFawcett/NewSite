LLM Architecture

How Large Language Models process text — a developer's walkthrough from input to output

01
Tokenization
like a compiler's lexer
+
Raw text is split into tokens — sub-word chunks from a fixed vocabulary (typically 32k–128k entries). This isn't word-splitting; it uses algorithms like Byte-Pair Encoding (BPE) that learn common character sequences from data. Each token maps to an integer ID. 
Input:  "Unbelievable!"

Step 1 — Tokenize:
  "Un"  "believ"  "able"  "!"

Step 2 — Map to IDs:
  [  1400,    25191,     481,     0  ]

Common words → 1 token       ("the" → 262)
Rare words   → many tokens   ("defenestrate" → "def" + "en" + "est" + "rate")
Code tokens  → "  " (2 spaces), "func", "return", etc.
Algorithm
Byte-Pair Encoding (BPE) or SentencePiece. Trained on the corpus to find optimal splits.
Dev analogy
Like a lexer turning source code into tokens — but for natural language, optimized for compression.
Why sub-words?
Handles any text (typos, new words, code) with a finite vocabulary. Zero out-of-vocabulary errors.
▼ token IDs
02
Embedding + Positional Encoding
like a hash map → dense vector
+
Each token ID becomes a high-dimensional vector (e.g., 4096 floats) via a learned lookup table. A positional encoding is added so the model knows token order — without this, "dog bites man" and "man bites dog" would be identical. 
Embedding Matrix (vocab_size × d_model)
┌──────────────────────────────────────────────┐
│  token 0   → [ 0.02, -0.15,  0.44, ... ]    │  d_model
│  token 1   → [-0.31,  0.08,  0.22, ... ]    │  = 4096
│  token 2   → [ 0.17,  0.59, -0.03, ... ]    │  floats
│  ...         (vocab_size rows)                │
└──────────────────────────────────────────────┘

token_embedding  =  embed_matrix[token_id]
final_embedding  =  token_embedding + position_encoding

Semantically similar words land near each other:
  cosine_sim("king", "queen")  ≈ 0.85
  cosine_sim("king", "banana") ≈ 0.12
Shape
Input: [batch, seq_len] ints → Output: [batch, seq_len, d_model] floats
Position encoding
Modern models use RoPE (Rotary Position Embeddings) — encodes position via rotation matrices. Generalizes to unseen sequence lengths.
Key insight
Embeddings are learned, not hand-designed. The model discovers that "king − man + woman ≈ queen" during training.
▼ vectors [seq_len × d_model]
03
Multi-Head Self-Attention
the core innovation
+
This is the heart of the Transformer. Each token computes how much to attend to every other token. It asks: "given what I represent, which other tokens in this sequence are relevant to me?" Three learned projections — Query, Key, Value — make this work. 
For each token, compute three vectors:

  Q = embedding × W_q     ← "What am I looking for?"
  K = embedding × W_k     ← "What do I contain?"
  V = embedding × W_v     ← "What info do I carry?"

Attention scores:

  scores = softmax( Q × Kᵀ / √d_k )

          "The"  "cat"  "sat"  "on"
  "The"  [ 0.1   0.4   0.3   0.2  ]   ← each row sums to 1
  "cat"  [ 0.2   0.1   0.5   0.2  ]   ← "cat" attends heavily to "sat"
  "sat"  [ 0.1   0.6   0.1   0.2  ]   ← "sat" attends heavily to "cat"
  "on"   [ 0.2   0.2   0.4   0.2  ]

Output:  output = scores × V       ← weighted mix of all token values

Multi-head (e.g., 32 heads run in parallel):
  Head 1 might learn: syntactic relationships (subject↔verb)
  Head 2 might learn: coreference ("it" → "the cat")
  Head 3 might learn: positional patterns (nearby words)
  ...then all heads are concatenated and projected back to d_model
Causal mask
During generation, token N can only attend to tokens 0..N. This prevents "seeing the future." Implemented as a triangular mask matrix.
Complexity
O(n²) in sequence length — this is why context windows are expensive. A 100k context = 10 billion attention computations per layer.
KV Cache
During generation, previously computed K and V are cached so each new token only computes one new row — not the full matrix. Critical optimization.
▼ context-enriched vectors
04
Feed-Forward Network (MLP)
like a per-token dense layer
+
After attention gathers context, each token passes independently through a 2-layer MLP. This is where the model does most of its "reasoning" — transforming the contextualized representation through learned non-linear functions. This is also where most of the model's stored knowledge (facts, patterns) lives. 
Per token (applied independently to each position):

  hidden  = activation_fn( x × W₁ + b₁ )    ← expand: d_model → 4×d_model
  output  = hidden × W₂ + b₂              ← compress: 4×d_model → d_model

Example dimensions (Llama 70B):
  d_model = 8192
  d_ffn   = 28672  (3.5× expansion with SwiGLU)

Residual connection + LayerNorm:

  x = LayerNorm( x + Attention(x) )
  x = LayerNorm( x + FFN(x) )
                 ↑
        skip connection — the original signal
        always passes through. The model learns
        to ADD refinements, not replace.
Activation
Modern LLMs use SwiGLU: SwiGLU(x) = Swish(xW₁) ⊙ xW₃. A gated variant that outperforms ReLU.
Parameter count
~⅔ of the model's total parameters are in FFN layers. This is the "memory bank" of facts and patterns.
Residual stream
Think of it as a shared bus — attention writes context, FFN writes transformations, all additive. Solves vanishing gradients.
▼ repeat N times
05
Stacked Transformer Layers
like a deep call stack
+
Stages 3 and 4 (attention + FFN) form one Transformer layer. Modern LLMs stack 32 to 128+ of these layers. Each layer refines the representation — early layers handle syntax and simple patterns, later layers handle semantics and complex reasoning. 
Layer Progression (what each depth "learns")

  Layer  0–10:  Surface patterns — POS tags, punctuation, local syntax
  Layer 10–30:  Semantics — word sense, entity types, relationships
  Layer 30–60:  Reasoning — multi-step inference, analogies, logic
  Layer 60–80:  Task execution — formatting, instruction following
  Layer 80+  :  Output shaping — final token prediction refinement

Residual stream view:

  input ──→ [+Attn₁] → [+FFN₁] → [+Attn₂] → [+FFN₂] → ... → [+AttnN] → [+FFNN] → output
           ↑_______________________________________________↑
            original signal persists through all layers (residual connections)
Typical depth
GPT-3: 96 layers. Llama 3 70B: 80 layers. Claude Sonnet-class: likely 60–100+. More layers = more "thinking steps."
Residual stream
The key abstraction: data flows through a "stream" and each layer adds a delta. Like middleware in a server stack — each one modifies the request.
▼ final hidden states
06
Output Head & Sampling
like a probability distribution → pick
+
The final hidden state of the last token is projected back to vocabulary-size logits, then converted to probabilities via softmax. The model picks the next token from this distribution — and the choice of sampling strategy dramatically affects output quality. 
Projection to vocabulary:

  logits = hidden_state × W_output      ← shape: [d_model] → [vocab_size]
  probs  = softmax( logits / temperature )

Example output distribution:
  "The cat sat on the ___"

  mat     ████████████████████  34%
  floor   ███████████████       22%
  couch   ██████████            15%
  bed     ██████                9%
  table   ████                  6%
  ...     (remaining ~50k tokens share the last 14%)

Sampling strategies:
  temperature=0   → always pick "mat"            (deterministic)
  temperature=0.7 → usually "mat" or "floor"     (balanced)
  temperature=1.5 → might pick "windowsill"      (creative)
  top_p=0.9       → sample from top 90% of mass  (nucleus sampling)
  top_k=50        → sample from top 50 tokens    (truncation)
Autoregressive loop
The chosen token is appended to the sequence, and the entire forward pass runs again for the next token. Generation is inherently sequential.
Speed bottleneck
Each token requires a full forward pass. A 1000-token response needs 1000 passes. KV caching makes each pass O(1) new computation.
Weight tying
W_output often shares weights with the embedding matrix (transposed). One matrix serves double duty, saving billions of parameters.

07
Training Pipeline
how all these weights are learned
+
Training happens in three major phases. First, massive self-supervised pre-training on internet-scale text. Then supervised fine-tuning on curated examples. Finally, RLHF/RLAIF to align the model with human preferences — this is what makes a raw text predictor into a useful assistant. 
Phase 1: Pre-training (weeks on thousands of GPUs)
┌─────────────────────────────────────────────────────────────┐
│  Objective: next-token prediction                          │
│                                                             │
│  Input:  "The capital of France is"                         │
│  Target: "Paris"                                            │
│  Loss:   cross_entropy(predicted_probs, "Paris")             │
│                                                             │
│  Data: ~10-15 trillion tokens (web, books, code, papers)    │
│  Cost: $10M–$100M+ in compute                              │
└─────────────────────────────────────────────────────────────┘
           ↓
Phase 2: Supervised Fine-Tuning (SFT)
┌─────────────────────────────────────────────────────────────┐
│  Train on curated (instruction, response) pairs             │
│  Teaches the model to follow instructions and be helpful    │
└─────────────────────────────────────────────────────────────┘
           ↓
Phase 3: RLHF / Constitutional AI
┌─────────────────────────────────────────────────────────────┐
│  1. Generate multiple responses to a prompt                 │
│  2. Human (or AI) ranks them: Response A > Response B       │
│  3. Train a reward model on these preferences               │
│  4. Use PPO/DPO to optimize the LLM against the reward     │
│                                                             │
│  This is what makes "raw GPT" → "ChatGPT"                  │
└─────────────────────────────────────────────────────────────┘
Optimizer
AdamW with learning rate warmup + cosine decay. Gradient checkpointing saves memory at the cost of recomputation.
Parallelism
Models are too large for one GPU. Training uses tensor parallelism (split layers), pipeline parallelism (split stages), and data parallelism (split batches).
Key insight
Pre-training learns capabilities. RLHF learns when and how to use them. The base model can write poetry — RLHF teaches it to do so when asked.
