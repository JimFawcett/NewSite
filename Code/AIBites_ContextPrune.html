AIBites: Context Pruning

managing context window size in Claude Code sessions

About Code Repo
"Prune early - a well-timed /compact is better than waiting for auto-compact."

1.0 - Introduction

A Claude Code context window fills with conversation history, file reads, tool results, CLAUDE.md content, memory, and MCP tool schemas. As the window fills, responses slow and cost rises. Three built-in commands give direct control over what stays and what goes: /clear, /compact, and /context. The right choice depends on what you are doing. Hard resets make sense between unrelated tasks. Structured summaries work within a task when you want to shed accumlated back-and-forth without losing intent. The context breakdown command tells you whether conversation history is actually the problem or whether the culprit is an oversized CLAUDE.md or too many active MCP servers.

2.0 - Core Commands

Three commands form the basic toolkit for pruning context: /clear - hard reset. Wipes the conversation entirely with no summary. CLAUDE.md files reload automatically from disk after the reset. Use this when switching to genuinely unrelated work - for example, jumping from a Rust tutorial repo to a website CSS fix. /compact - replaces conversation history with a structured summary rather than deleting it. You can steer it: /compact focus on the auth bug fix keeps what you choose instead of what the automatic pass guesses matters. Use it proactively - before the window gets crowded - rather than waiting for auto-compact, because Claude summarizes better with full recall and more headroom. /context - shows a live breakdown of what is consuming your context window: CLAUDE.md size, memory, MCP tool schemas, skills, and conversation. Includes optimization suggestions. Run this first when a session feels sluggish - sometimes the problem is not conversation history at all but a bloated CLAUDE.md or too many MCP servers.
When to use each command
  1. Run /context to identify what is actually consuming the window. If CLAUDE.md or MCP schemas are the problem, /clear or /compact will not help much - address the source instead.
  2. Use /compact focus on <topic> between phases of a long task - for example, between spec-writing and implementation - to carry intent forward without carrying every intermediate exchange.
  3. Use /clear when starting a completely different task in the same terminal session. CLAUDE.md reloads automatically so project context is restored.

3.0 - What Survives /compact

Not everything is equivalent after a /compact. Some project context reloads from disk automatically; some requires touching the right file to restore. Reloaded automatically from disk:
  • Project-root CLAUDE.md and any unscoped rules
  • Auto memory files
 Lost until a file is touched again:
  • Rules with paths: frontmatter - lost until a file matching that path pattern is read. To keep a rule active across compactions, remove the paths: frontmatter or move the rule into root CLAUDE.md.
  • Nested CLAUDE.md files in subdirectories - lost until a file in that subdirectory is touched again.
Skill bodies: reloaded but capped at 5,000 tokens per skill and 25,000 tokens total. Oldest skills are dropped first when the cap is reached. Put critical instructions near the top of the skill definition to ensure they survive.
Implications for CLAUDE.md hierarchies
  1. If you use path-scoped rules (paths: ["src/auth/**"] in frontmatter), plan for them to be absent after a compact. Either restructure them as unscoped rules in root CLAUDE.md or ensure the first action after compacting reads a file in the relevant directory.
  2. Nested CLAUDE.md files (for example, a Learn/CLAUDE.md) are transparent after compact until a file under that directory is accessed. If you use phase-based compaction, prime the subdirectory explicitly early in each new phase.
  3. For skill-heavy setups, order critical instructions at the top of each skill file - they survive longest when the 25,000-token cap forces drops.

4.0 - Other Levers

Two additional tools address context size without requiring a full reset or compact: Subagents - spawning a subagent keeps large file reads entirely out of the main context. Only the final summary comes back. This is useful before deciding what to change in a large codebase: delegate the broad exploration to a subagent, review the summary, then act in the main session. /rewind - undoes recent turns without leaving a trail of corrective back-and-forth in history. Preferable to writing "no, actually do X instead" messages that themselves consume context and add noise to any future compact summary.
