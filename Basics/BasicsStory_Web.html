Chapter #7 – The Web

HTTP evolution, browser pipeline, HTML/CSS/JS, architecture, APIs, performance, security

7.0  Prologue

The web is an application layer built on the internet - a system of interconnected resources identified by URLs and exchanged via HTTP. What started as a document-sharing system at CERN in 1989 has evolved into the dominant platform for delivering interactive applications indistinguishable from native software. The internet carries packets; the web assigns meaning to them. Understanding how the web works - from the HTTP wire protocol to the browser rendering pipeline - is foundational for writing software that delivers reliable, fast, and secure user experiences.

7.1  HTTP Protocol Evolution

HTTP has passed through five generations, each addressing limitations exposed by the previous version at production scale:
Version Year Transport Key Changes
HTTP/0.9 1991 TCP Single-line GET only; HTML response; no headers or status codes.
HTTP/1.0 1996 TCP Headers, status codes, multiple content types; one TCP connection per request.
HTTP/1.1 1997 TCP Persistent connections, pipelining, chunked transfer, virtual hosting via Host header. Head-of-line blocking within a connection.
HTTP/2 2015 TCP + TLS Binary framing; multiplexed streams on one connection; HPACK header compression; server push. TCP-level head-of-line blocking remains.
HTTP/3 2022 QUIC (UDP) Built on QUIC; eliminates TCP head-of-line blocking; 0-RTT resumption; connection migration across IP changes.
Head-of-line blocking is the central problem HTTP/2 and HTTP/3 address in different ways. In HTTP/1.1, a slow response blocks all subsequent requests on the same connection. HTTP/2 multiplexes requests as independent streams on one TCP connection, so a slow response does not block other streams - but a lost TCP segment stalls all streams until the segment is retransmitted, because TCP guarantees ordered delivery. HTTP/3 runs on QUIC, which implements independent streams inside UDP datagrams; a lost packet stalls only the stream it belongs to, not all streams. QUIC integrates TLS 1.3 into its handshake, reducing connection setup from two round trips (TCP SYN + TLS) to one. Subsequent connections from a known server can use 0-RTT resumption to send application data with the first packet, eliminating handshake latency entirely. Connection migration allows a client to change IP addresses - switching from Wi-Fi to cellular, for example - without interrupting the session, because QUIC identifies connections by a connection ID rather than the source IP and port.

7.2  Browsers and the Rendering Pipeline

A browser converts bytes received over HTTP into a rendered, interactive page through a fixed pipeline of stages:
  1. Navigation: DNS resolution, TCP/TLS connection, HTTP request. The browser may use cached DNS results, reuse an existing connection, and send conditional headers (If-None-Match, If-Modified-Since) to allow 304 Not Modified responses.
  2. HTML parsing and DOM construction: the HTML parser builds a tree of nodes (the DOM) incrementally as bytes arrive. Parsing pauses when the parser encounters a synchronous <script> tag because script can call document.write() and alter the document structure.
  3. CSS parsing and CSSOM construction: stylesheets are render-blocking. The browser will not render content until all linked stylesheets are downloaded and parsed into the CSS Object Model. This prevents a flash of unstyled content but delays first paint when stylesheets are large or slow to arrive.
  4. Render tree construction: the DOM and CSSOM are merged into a render tree containing only visible nodes (elements with display: none are excluded). Each node holds computed style information.
  5. Layout (reflow): the browser computes the exact geometry - position and size - of each render tree node relative to the viewport. Layout is triggered again whenever DOM or styles change in a way that affects geometry.
  6. Paint: the browser fills pixels into layers. Text, borders, shadows, and backgrounds are drawn in a defined painting order.
  7. Compositing: the GPU assembles layers into the final frame. Layers promoted to the GPU (via transform or will-change) can be moved or faded without triggering layout or paint, enabling smooth animations.
Script loading strategies control when JavaScript runs relative to HTML parsing:
  • Default (synchronous): parser pauses, script downloads and executes, then parsing resumes. Blocks first paint for scripts in <head>.
  • defer: script downloads in parallel with parsing, executes after the DOM is complete but before the DOMContentLoaded event. Execution order among deferred scripts is preserved. Preferred for scripts that need the full DOM.
  • async: script downloads in parallel and executes immediately when loaded, potentially before parsing finishes. Order is not guaranteed. Suited for independent third-party scripts such as analytics.
JavaScript engines (V8 in Chrome/Node.js, SpiderMonkey in Firefox, JavaScriptCore in Safari) use just-in-time compilation. Frequently executed code is recompiled from bytecode to optimized machine code based on runtime type observations. If a type assumption is later invalidated, the engine deoptimizes and falls back to interpreted bytecode.

7.3  HTML, CSS, and JavaScript

HTML5 introduced semantic elements that carry meaning beyond layout: <header>, <nav>, <main>, <article>, <section>, <aside>, and <footer>. These help screen readers and search engines understand page structure without relying on CSS class names. HTML5 also standardized <canvas> for 2D and WebGL graphics, <audio> and <video> for native media, and the <template> element for inert DOM fragments used in component patterns. CSS controls visual presentation through a cascade of rules applied to the DOM. Key layout systems:
  • Flexbox: one-dimensional layout; distributes space along a main axis with optional wrapping. Suited for navigation bars, card rows, and aligned form controls.
  • CSS Grid: two-dimensional layout; defines rows and columns explicitly; items are placed into named grid areas. Suited for full-page layouts and complex component grids.
  • CSS custom properties (variables): --token: value declared in :root and consumed via var(--token). Enables theming without preprocessors.
  • Media queries: apply rule sets conditionally based on viewport width, orientation, or display type; the foundation of responsive design.
JavaScript is the only scripting language browsers execute natively. Key language characteristics relevant to web development:
  • Event loop: JavaScript is single-threaded. The event loop processes a task queue; each task runs to completion before the next begins. Microtasks (Promise callbacks) drain completely after each task before the next macrotask (setTimeout, I/O callbacks) starts.
  • Promises and async/await: non-blocking I/O is expressed as Promises. async functions return a Promise; await suspends execution within the function until the Promise resolves, without blocking the event loop.
  • ES Modules (ESM): import/export syntax enables static analysis of dependencies. Browsers load modules with deferred semantics; bundlers (Vite, esbuild, Webpack) merge modules for production.
  • WebAssembly (Wasm): a binary instruction format compiled from C, C++, or Rust that runs in the same browser sandbox at near-native speed. Wasm modules are imported into JavaScript and called as functions; they cannot access the DOM directly but pass data through JavaScript interop.

7.4  Web Application Architecture

How an application divides work between server and client determines its performance profile, SEO behavior, and infrastructure requirements:
Pattern Rendering location First paint Notes
SSR (Server-Side Rendering) Server, per request Fast Full HTML in first response; works without JS; requires a running server for every request.
CSR (Client-Side Rendering) Browser, via JS bundle Slow Rich interactivity; static host suffices; poor SEO without additional measures; JS bundle must load before rendering.
SSG (Static Site Generation) Build time Fastest Pre-rendered HTML served from CDN; no server at runtime; stale for dynamic data.
ISR / Edge rendering CDN edge, on demand Fast Pages revalidated at intervals or on demand; combines SSG performance with fresh data.
Single-Page Applications (SPAs) load once and update the DOM in place as the user navigates. The History API (pushState, replaceState) allows URL changes without a full page reload. Frameworks like React, Vue, and Svelte manage a virtual DOM or compiled reactive graph that diffs desired state against the current DOM and applies minimal updates. API patterns define how the browser fetches data from the server:
  • REST: resources at stable URLs; HTTP methods convey intent; stateless. Covered in Chapter 6, section 6.3.
  • GraphQL: the client specifies exactly which fields it needs in a query; the server returns only those fields. Eliminates over-fetching (too many unwanted fields) and under-fetching (too few fields requiring follow-up requests).
  • WebSockets: full-duplex persistent connection over a single TCP connection, initiated by an HTTP upgrade handshake. Used for real-time features: chat, live dashboards, collaborative editing.

7.5  Web APIs and Modern Standards

Browsers expose a large set of APIs beyond DOM manipulation. Key ones for application developers:
  • Fetch API: promise-based HTTP client built into the browser. Replaces XMLHttpRequest with a cleaner interface; supports streaming response bodies and AbortController for request cancellation.
  • Service Workers: a JavaScript file installed by the browser as a proxy between the page and the network. Service workers intercept fetch events and can serve cached responses, enabling offline operation. They run in a separate thread with no DOM access and communicate with pages via postMessage. They are the foundation of Progressive Web Apps (PWAs).
  • Web Workers: background computation threads that run JavaScript off the main thread. They cannot access the DOM but perform CPU-intensive tasks (image processing, cryptography, data parsing) without blocking the UI. Data is exchanged by structured clone via postMessage.
  • WebRTC (Real-Time Communication): peer-to-peer audio, video, and data channels between browsers without a relay server for media. ICE (Interactive Connectivity Establishment) negotiates a path through NAT using STUN servers; if direct connection fails, a TURN server relays traffic.
  • IndexedDB: a transactional key-value database in the browser for structured data persistence across sessions. Larger storage quota than localStorage; asynchronous; supports indexes and range cursors.
  • WebAssembly System Interface (WASI): a standardized set of system call abstractions for Wasm modules running outside the browser (in runtimes such as Wasmtime and WasmEdge). Enables portable server-side Wasm deployable at CDN edges.

7.6  Web Performance

Google's Core Web Vitals define measurable performance targets that correlate with user experience and influence search rankings:
Metric Measures Good threshold
LCP (Largest Contentful Paint) Loading - time until the largest visible content element is rendered. ≤ 2.5 s
INP (Interaction to Next Paint) Responsiveness - time from user interaction to next frame update. ≤ 200 ms
CLS (Cumulative Layout Shift) Visual stability - sum of unexpected layout shifts during page life. ≤ 0.1
Critical rendering path optimization:
  • Inline the minimal CSS needed for above-the-fold content; load the rest asynchronously to unblock first paint.
  • Use defer on non-essential scripts; move render-blocking scripts to end of <body> when defer is not viable.
  • Preload critical assets with <link rel="preload"> so the browser fetches them at highest priority before the parser discovers them.
  • Establish early connections with <link rel="preconnect">, which pays the DNS + TCP + TLS cost before the request is needed.
Caching strategies:
  • Cache-Control: max-age=N: browser serves the cached response for N seconds without contacting the server.
  • Cache-Control: stale-while-revalidate=N: browser serves stale content immediately while fetching a fresh copy in the background, eliminating the revalidation round trip from the user's critical path.
  • Content-hashed file names: bundlers embed a hash of file contents in the filename (e.g., main.3f8a2b.js). These files can be cached forever (max-age=31536000, immutable); the HTML document references the new hash when content changes.
  • Service worker cache: full programmatic control; implements cache-first, network-first, or stale-while-revalidate strategies per resource type, enabling reliable offline operation.
Code splitting and lazy loading reduce initial bundle size. A dynamic import('./module.js') call defers loading a module until it is needed, splitting the application into chunks loaded on demand. Images with loading="lazy" are deferred until they approach the viewport.

7.7  Web Security in Practice

Chapter 6 covers network-level threats (XSS, CSRF, injection, TLS). This section focuses on the browser security model and the HTTP mechanisms that enforce it. Same-Origin Policy (SOP): the browser's foundational security rule. A script from origin A (scheme + host + port) cannot read resources from origin B unless B explicitly grants permission. This prevents a malicious page from reading a user's bank account data using an active authenticated session. CORS (Cross-Origin Resource Sharing): the controlled relaxation of SOP. A server opts in to cross-origin access via response headers:
  • Access-Control-Allow-Origin: https://app.example.com - permits requests from a specific origin.
  • Preflight: for non-simple requests (PUT, DELETE, or custom headers), the browser sends an OPTIONS preflight to confirm the server allows the actual request before sending it.
CORS is enforced by the browser, not the server. A server can claim any CORS headers, but the browser is the entity protecting the user's data. Content Security Policy (CSP): an HTTP response header that whitelists which sources the browser may load scripts, styles, images, and frames from. A strict CSP eliminates most XSS attack surface by preventing inline script execution and restricting external script origins: Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com Cookie security attributes:
  • HttpOnly - cookie is inaccessible to JavaScript; prevents session theft via XSS.
  • Secure - cookie is transmitted only over HTTPS connections.
  • SameSite=Strict - cookie is never sent on cross-site requests, blocking CSRF entirely. Lax allows it on top-level navigations.
Subresource Integrity (SRI): when loading scripts or stylesheets from a CDN, the integrity attribute carries a Base64-encoded hash of the expected file contents. The browser verifies the hash before executing the resource; a tampered CDN file fails the check and is blocked: <script src="https://cdn.example.com/lib.js" integrity="sha384-oqVu..." crossorigin="anonymous"></script> HTTPS and HSTS: TLS authenticates the server and encrypts all traffic. The Strict-Transport-Security header instructs the browser to refuse plain HTTP connections to the domain for a specified duration (typically one year). Domains on the HSTS preload list are hardcoded into browsers, so a first visit is already protected without an initial HTTP redirect.

7.8  Epilogue

The web is built from a small set of composable technologies - HTTP, HTML, CSS, and JavaScript - that have been incrementally extended without breaking backward compatibility. The browser security model, performance tooling, and server architecture patterns all exist to manage the tension between that openness and the demands of production software. The next chapter examines the data centers that host the server side of these interactions.
