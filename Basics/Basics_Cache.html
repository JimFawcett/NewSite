Basic Bites: Cache Lines

Cache hierarchy, coherence, visibility, false sharing, synchronization

About Track Summary
A cache line is the smallest unit of data the CPU moves between main memory and its caches. On every modern x86-64 and ARM processor the cache line is 64 bytes. When a core reads one byte, the hardware loads the entire 64-byte block containing that byte. When a core writes one byte, the hardware must own the entire 64-byte block before the write completes. This block-level ownership governs both performance and correctness in multi-core programs.

1.  Cache Hierarchy

Modern processors place several levels of cache between the CPU registers and DRAM. Each level is larger, slower, and shared by progressively more cores:
Level Typical size Typical latency Sharing scope
L1-I / L1-D 32–64 KB each 4–5 cycles (~1 ns) private per core
L2 256 KB–2 MB 12–15 cycles (~4 ns) private per core (or per two cores)
L3 (LLC) 8–96 MB 30–50 cycles (~15 ns) shared across all cores on the die
DRAM 16 GB–1 TB ~200–300 cycles (~80 ns) shared across all cores
A cache hit means the line was already present at the requested level; a cache miss means the hardware must fetch it from the next level. An L1 hit costs a handful of cycles; a DRAM miss costs hundreds. Programs that keep their working set in L1 or L2 run an order of magnitude faster than programs that thrash DRAM.

2.  Cache Coherence

Each core has its own private L1 and L2 caches, so the same cache line can reside simultaneously in multiple caches. When one core modifies its copy, all other copies must be invalidated or updated; otherwise cores would disagree on the value of a memory location. This guarantee is called cache coherence, and it is enforced in hardware by a protocol, not by software. The dominant protocol on x86-64 is MESI, which assigns each cache-line slot one of four states:
State Meaning
Modified This core holds the only valid copy and has written to it; DRAM is stale.
Exclusive This core holds the only copy; it matches DRAM and no other core has it cached.
Shared One or more cores hold clean copies; DRAM is current.
Invalid This cache-line slot is stale and must be reloaded before use.
When core A writes to a line it holds in Modified or Exclusive state, the hardware issues an invalidation on the interconnect. Every other core holding that line in Shared state transitions it to Invalid. The next read on any of those cores triggers a miss and a fetch of the updated data. This means that a write on one core eventually becomes visible to all others — but eventually is the key word; visibility is not instantaneous.

3.  Visibility

Coherence guarantees that caches will not permanently disagree, but it does not guarantee when a write becomes visible to another core. Two additional sources of reordering delay visibility in software: Store buffers. Before a core commits a write to its L1 cache, the write enters a hardware store buffer. The same core may read its own buffered write (store forwarding), but other cores cannot see it until it drains into L1 and the coherence protocol propagates it. The store buffer depth is not architecturally visible to software. Compiler and CPU reordering. Both the compiler and the out-of-order execution engine may reorder loads and stores relative to their program-order positions, provided the reordering is invisible to the executing core. From another core's perspective, writes can appear to arrive in a different order than the program performed them. Memory fences (barriers). A fence instruction — x86 MFENCE, ARM DMB — forces the CPU to drain its store buffer and prevents reordering of loads and stores across the fence point. The compiler must also be prevented from reordering around it, typically via an intrinsic that acts as a compiler barrier. Together they establish a happens-before edge: every write before the fence is visible to any core that executes a corresponding fence before its subsequent loads. In the C++ and Rust memory models, atomic operations with Release and Acquire ordering map directly to these fences without requiring inline assembly. A store(Release) on one thread paired with a load(Acquire) on another establishes a synchronization point: all writes before the store are guaranteed visible after the load.

4.  False Sharing

Because coherence operates at cache-line granularity, two threads can interfere with each other even when writing to different variables, if those variables happen to share a cache line. Core A writes to byte 0; Core B writes to byte 8. Neither reads or writes the other's variable, but the coherence protocol treats the entire 64-byte line as the unit of ownership. Each write forces the other core to re-acquire exclusive ownership of the whole line before it can proceed. This is false sharing: the cores appear to contend on shared data, but the contention is an artifact of memory layout, not program logic. Performance impact. False sharing converts independent writes into a cache-line ping-pong across the core interconnect. Each round-trip costs 40–100 ns on a typical chip; on a multi-socket NUMA system it can exceed 200 ns. Throughput can drop by an order of magnitude compared to a version where each thread writes its own private line. Eliminating false sharing. Pad or align contested variables so that each occupies its own cache line:
  • C++17: alignas(std::hardware_destructive_interference_size) on the struct member or type.
  • Rust: #[repr(align(64))] on a newtype wrapper, or the crossbeam_utils::CachePadded<T> wrapper.
  • C#: explicit field offsets via [StructLayout(LayoutKind.Explicit)], or manual padding fields totaling 64 bytes per slot.

5.  True Sharing and Synchronization

When threads intentionally share a variable — one writes, another reads — cache coherence ensures the reader eventually sees the update, but without synchronization primitives the compiler or CPU may reorder surrounding instructions so that the reader observes a partially constructed value or an indefinitely stale one. Two tools address this: Atomic operations. An atomic read-modify-write is a single indivisible hardware operation on one naturally aligned word (up to 16 bytes on x86-64). It eliminates torn reads and torn writes and carries a memory-order parameter that constrains reordering of surrounding loads and stores. Atomics do not lock a cache line in the sense of blocking other cores; instead they rely on hardware primitives such as LOCK CMPXCHG (x86) or LDXR/STXR (ARM), which implicitly generate the coherence traffic needed to obtain exclusive ownership of the line. Mutexes and memory ordering. A mutex lock/unlock pair contains implicit full memory fences. Every write completed before unlock() is guaranteed visible to the thread that next succeeds at lock(). This is stronger than a plain fence: it also provides mutual exclusion so that at most one thread at a time observes a consistent snapshot of the guarded data. Volatile is not sufficient. In C++, volatile prevents the compiler from caching a variable in a register, but it does not emit memory fences and does not prevent CPU reordering. On x86-64 it sometimes appears to work because the x86 memory model is relatively strong, but on ARM or PowerPC it provides no ordering guarantee. Use std::atomic for inter-thread communication; reserve volatile for memory-mapped I/O registers.

6.  Language Support

Language Atomic type Cache-line alignment
C++ std::atomic<T> alignas(std::hardware_destructive_interference_size) (C++17; 64 on x86-64 and ARM64)
Rust std::sync::atomic::AtomicUsize, AtomicBool, etc. #[repr(align(64))] on the containing struct, or crossbeam_utils::CachePadded<T>
C# Interlocked class methods; volatile keyword (compiler fence only — not sufficient on non-x86) [StructLayout(LayoutKind.Explicit)] with explicit field offsets, or manual padding fields totaling 64 bytes per slot
Python The GIL serializes CPython threads; no cache-line atomics are exposed. Use multiprocessing for true parallelism — separate processes have separate address spaces and no shared cache lines. Not applicable under the GIL; manual alignment possible via ctypes

7.  Consequences

Cache-line behavior has direct consequences for how concurrent code is designed and tuned:
  • Struct layout matters. Fields written by different threads must not share a cache line. Group read-only fields together; separate hot write fields with explicit padding.
  • Contiguous data structures are cache-friendly. Arrays keep sequential elements in a small number of lines. Linked lists, trees, and pointer-chasing graphs scatter nodes across lines, producing a miss on every traversal step.
  • Read-mostly data needs no synchronization. Multiple cores can hold the same line in Shared state simultaneously without invalidating each other. Immutable data avoids all coherence traffic.
  • Atomic operations are not free. A counter incremented by every thread on every iteration is a serialization bottleneck: the line must be acquired exclusively on each increment. Use per-thread counters and aggregate at the end.
  • NUMA amplifies the cost. On multi-socket systems, accessing memory attached to a remote NUMA node adds another 100–200 ns per miss. False sharing across NUMA nodes is especially expensive; keep each thread's working data on its local node.
