# Edit.md — Writing Style Guide

## Audience

Experienced software professionals. Skip basics. Use precise technical terms without definition. Trust the reader.

## Voice

- First person when describing personal work or opinions: "I built this to...", "I think...", "I expect..."
- Address the reader directly as "you" where natural
- Conversational but not casual — collegial, not chatty
- Honest about limitations and work-in-progress state
- Short, declarative sentences. One idea per sentence
- Avoid hedging phrases like "it should be noted", "it is worth mentioning", "please note"

## Active Voice

Write in active voice. The subject acts; it is not acted upon.

- Prefer: "The compiler enforces ownership rules."
- Avoid: "Ownership rules are enforced by the compiler."

## Punctuation

- No em dashes. Replace with a period, comma, semicolon, or restructure the sentence
- Use commas for brief parenthetical asides
- Use a new sentence when the aside is substantive

## Tone

- Direct. State what something does, not what it "allows you to" or "enables you to" do
- Prefer: "Rust prevents data races at compile time."
- Avoid: "Rust allows you to prevent data races at compile time."
- Acknowledge tradeoffs plainly: "C++ is fast but requires careful memory management."
- Use "at this time" or "eventually" rather than vague phrases like "in the future" or "soon"

## Structure

- Use bullet lists for enumeration, not long compound sentences
- Use short paragraphs — two to four sentences each
- Lead with the main point; supporting detail follows

## Things to Avoid

- Em dashes
- Passive constructions
- Filler phrases: "essentially", "basically", "in order to", "it is important to note"
- Redundant qualifiers: "very", "quite", "really", "fairly"
- AI-sounding constructions: "delve into", "leverage", "robust", "seamlessly", "harness the power of"
